## Блок 1. Контекст и проблема

### 1.1 Контекст инициативы
**Проект:** «Мера социальной поддержки»  
**Суть:** Поддержка в актуальном состоянии исчерпывающего перечня мер социальной поддержки для трёх категорий граждан РФ:
- Люди с инвалидностью
- Ветераны боевых действий
- Участники СВО

Источник — нормативно-правовые акты и официальные государственные источники.

### 1.2 Проблемная область
Актуальность данных поддерживается **вручную**. Это трудозатратно, медленно и чревато ошибками. Изменения в законодательстве происходят регулярно (индексации выплат, новые постановления, утрата силы старых), и ручной мониторинг не поспевает.

### 1.3 Последствия бездействия
- **Ошибки в данных:** неверные суммы, пропущенные/устаревшие меры поддержки
- **Репутационные потери:** граждане получают недостоверную информацию
- **Жалобы и обращения:** от пользователей, которым система выдала неверные данные
- **Потеря времени:** ручная перепроверка каждого НПА отнимает часы еженедельно
- **Юридические риски:** неактуальные ссылки на отменённые НПА
- **Упущенные меры:** новые выплаты/льготы не попадают в перечень вовремя

## Блок 4. Решение

### 4.1 Ключевые функциональные возможности

**Must have (все три):**

| # | Функция | Приоритет | Описание |
|---|---------|-----------|----------|
| F1 | **Поиск НПА** | Must have | Поиск нормативных документов по мерам соцподдержки (федеральный и региональный уровни) по заданным категориям |
| F2 | **Анализ и извлечение** | Must have | Извлечение информации в структурированном формате по параметрам; перевод нормативного языка на общедоступный |
| F3 | **Актуальность и отслеживание изменений** | Must have | Проверка актуальности, ежегодные индексации выплат, мониторинг изменений законодательства |

### 4.2 Пользовательские сценарии

**Happy path (типовой день Ирины):**

1. Ирина открывает **дашборд**
2. Видит три базы данных по категориям (инвалиды, ветераны, СВО)
3. Запускает **сверку** базы по выбранной категории с действующими НПА
4. Далее — автоматически по расписанию: поиск обновлений за прошедший период

**Что делает система:**
- Ищет изменения в НПА
- Анализирует и извлекает меры с параметрами (название, тип, сумма, получатель, условия, сроки, основание)
- Сверяет с текущей базой
- Помечает **новые** и **изменившиеся** меры

**Результат на дашборде Ирины:**
- Сводка: «Найдено N новых мер, M изменённых»
- По каждой мере — карточка с параметрами и ссылкой на пункт документа
- По изменённой — подсветка: что именно поменялось
- Если параметр не найден — честно: «Не указано в документе»

**Завершение:** Ирина проверяет, жмёт «Подтвердить» → база обновлена → задача закрыта.

### 4.3 Граничные случаи и исключения

| Сценарий | Действие системы |
|----------|------------------|
| **Поиск: документ не найден/недоступен** | Сообщает: «Недоступен. Проверьте ссылку или вручную». Даёт поле для ручной загрузки |
| **Анализ: документ не парсится** | Предупреждение с пометкой «Проверить вручную» |
| **Актуальность: источник недоступен** | 3 попытки → сообщение: «Источник временно недоступен, задача отложена» |
| **Актуальность: документ утратил силу** | Ищет замену. Если нет — помечает меры как недействующие, не удаляет, ставит «Документ утратил силу» + дата утраты силы |
| **Актуальность: мера отменена новым НПА** | Подсвечивает «Исключена» + дата исключения |
| **Актуальность: истёк срок действия меры** | «Срок истёк» + дата завершения |
| **Сверка: мера исчезла из нового НПА** | Подсвечивает. Не удаляет. Ирина решает: пометить недействующей или оставить с комментарием |
| **Сверка: параметр изменился** | Сравнение «было → стало» со ссылками на оба НПА, ждёт подтверждения |
| **Сверка: похоже на дубликат** | Показывает карточки рядом. Ирина выбирает: объединить, заменить или оставить обе |
| **Сверка: противоречие между документами** | Подсветка красным, показывает источники, **требует ручного разрешения** |

**Главные правила:**
1. Автоматически ничего не удаляем — только по команде Ирины
2. Частичный результат отдаём, а не отбрасываем
3. О любой проблеме говорим прямо и предлагаем действие
4. Все изменения логируем: что, когда и на основании какого документа поменялось

## Блок 7. Риски и митигации

### 7.1 Идентифицированные риски

| # | Риск | Категория | Вероятность | Влияние | Приоритет |
|---|------|-----------|-------------|---------|-----------|
| R1 | **Качество парсинга НПА** — разные форматы (PDF, скан, вёрстка), не всё парсится корректно | Технический | Высокая | Высокое | Критический |
| R2 | **Доступность источников** — официальные сайты могут быть недоступны, менять URL без предупреждения | Технический | Средняя | Среднее | Средний |
| R3 | **Частота изменений** — неравномерная: всплески индексаций, новые законы | Бизнес | Средняя | Среднее | Средний |
| R4 | **Adoption** — Ирина привыкла к ручному процессу, может не доверять автоматизации | Организационный | Средняя | Среднее | Средний |
| R5 | **Сроки** — MVP за 5 дней, релиз за 16 дней, возможны задержки | Организационный | Высокая | Критическое | Критический |

### 7.2 Митигации

| Риск | Митигация |
|------|-----------|
| **R1. Качество парсинга** | — Fallback: ручная загрузка документа при неудаче<br>— Пометка «Проверить вручную» для непрозрачных случаев<br>— Постепенное обучение на реальных документах |
| **R2. Доступность источников** | — 3 попытки с задержкой<br>— Поле для ручной загрузки<br>— Логирование всех сбоев для анализа |
| **R3. Частота изменений** | — Гибкое расписание проверок<br>— Приоритизация категорий по частоте изменений |
| **R4. Adoption** | — Прозрачность: система показывает источники каждого решения<br>— Режим «только рекомендация» на старте<br>— Ирина всегда подтверждает изменения вручную |
| **R5. Сроки** | — MVP: минимальный набор функций (F1 + F2, без полноценного F3)<br>— Приоритет: работающий прототип > полная функциональность<br>— План отката: вернуться к ручному процессу при срыве сроков |
