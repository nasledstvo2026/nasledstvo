# HCM — Hybrid Classifier for Multiagent Router

**Цель:** Устранить провал Router-агента на Knowledge-only запросах (MMLU benchmark: 11% против 80% Plain LLM) через внедрение классификации запросов до выбора цепочки агентов.

---

## 1. Классы запросов

| Класс | Описание | Примеры | Маршрут |
|---|---|---|---|
| **K** (Knowledge-only) | Ответ известен модели из обучения. Поиск внешних данных ухудшает точность (шум, устаревшие данные). | «Какой год основания Римской империи?», «Определение теории относительности», «Кто такой Фрейд?» | Router → Solver (минуя Collector и Verifier) |
| **T** (Tool-use) | Требует внешних данных: свежие новости, законы, цены, документы. | «Какая ставка ЦБ РФ сегодня?», «Последняя редакция 44-ФЗ», «Стоимость доллара» | Router → Collector → Verifier → Solver |
| **H** (Hybrid) | Нужны и факты из памяти, и внешняя верификация. | «Сравнить налоги в РФ и Германии в 2024», «Применима ли эта статья к моему случаю?» | Router → Collector (для сверки) → Verifier → Solver |

### Дополнительно: Borderline
Запрос, который Router не может уверенно классифицировать:
→ Маршрут **T** по умолчанию (Tool-use, консервативная стратегия). Верификация надёжнее потери точности.

---

## 2. Архитектура: два варианта

### Вариант A: Классификация внутри Router (рекомендуемый)
```
User → Router (с классификатором)
         ├─ K → Solver (ответ из LLM)
         ├─ T → Collector → Verifier → Solver
         └─ H → Collector → Verifier → Solver
```
**Плюсы:** Минимум изменений, один лишний шаг в промпте Router.
**Минусы:** Router тратит токены на классификацию; нет отдельного промпта классификатора.

### Вариант B: Вынос в Classifier-agent
```
User → Classifier (агент-классификатор)
         ├─ K → Router → Solver
         ├─ T → Router → Collector → Verifier → Solver
         └─ H → Router → Collector → Verifier → Solver
```
**Плюсы:** Чистая ответственность, можно тюнить классификатор отдельно.
**Минусы:** Дополнительный вызов LLM (латентность + стоимость), Router всё ещё нужен для маршрутизации.

**Вердикт:** Вариант A. Router уже существует — добавить в его промпт блок классификации дешевле и быстрее.

## 3. Промпт Router с классификацией

### Системный промпт Router (дополнение)

```
## Классификация запроса

Перед маршрутизацией определи КЛАСС запроса по приоритету:

### K (Knowledge-only) — НЕ ИСПОЛЬЗУЙ ИНСТРУМЕНТЫ
Запрос満足ляет ОДНОМУ из критериев:
1. Академический/исторический факт: даты, события, персоналии, термины
2. Концептуальный/теоретический вопрос: определение, объяснение явления
3. Объективный факт из общеизвестного знания (школьная/вузовская программа)
4. Вопрос не содержит отсылок к текущей дате, законам, документам, ценам
→ Ответь сразу из памяти модели. Маршрут: → Solver

### T (Tool-use) — ИСПОЛЬЗУЙ ИНСТРУМЕНТЫ
Запрос満足ляет ЛЮБОМУ из критериев:
1. Требует свежих данных: новости, курсы, погода, события {current_date}
2. Ссылается на конкретный закон/НПА: «44-ФЗ», «ст. 152 ГК», «Постановление №...»
3. Содержит «проверь», «найди», «посмотри», «узнай», «актуально ли»
4. Требует документа/справки/выписки
→ Используй Collector для поиска. Маршрут: → Collector → Verifier → Solver

### H (Hybrid) — ИСПОЛЬЗУЙ ИНСТРУМЕНТЫ
Запрос満足ляет ВСЕМ критериям:
1. Содержит факты, которые можно извлечь из модели
2. И содержит запрос на верификацию или сравнение с внешними данными
→ Используй Collector для сверки. Маршрут: → Collector → Verifier → Solver

### Borderline (неуверен)
Если не можешь уверенно определить класс:
→ Маршрут T по умолчанию (безопасная сторона)
```

### Примеры маршрутизации

| Запрос | Класс | Почему |
|---|---|---|
| «Какие три закона Ньютона?» | K | Школьная программа, не нужны внешние данные |
| «Последняя ставка ЦБ РФ» | T | Нужен внешний источник (обновляется) |
| «Сравни КоАП РФ и КоАП РБ по срокам давности» | H | Содержит знание (что такое КоАП) + сверку редакций |

## 4. Ожидаемое влияние на benchmark

| Benchmark | Текущий (без классификации) | После внедрения | Комментарий |
|---|---|---|---|
| MMLU (Knowledge) | **~11%** | **~70-80%** | K-запросы пойдут сразу в Solver |
| Tool-use (поиск законов) | ~85% | ~85% | Без изменений |
| Hybrid (сравнение+верификация) | ~60% | ~65-70% | Чёткое понимание когда нужна сверка |
| Общая точность | ~50% | ~70-75% | |

### Оценка затрат
- **Token cost:** +50-100 токенов на запрос (классификация в Router)
- **Latency:** +0.1-0.2 сек на классификацию
- **Экономия:** -1 полноценный вызов Collector для K-запросов (~500-2000 токенов + поиск)
