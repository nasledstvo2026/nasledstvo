## 1. Предпосылки

Успешный опыт архитектуры Катрин (двухагентная схема с изоляцией и верификацией) используется как архитектурный прототип.

На момент старта продукт рассчитан на одного пользователя, но архитектура позволяет масштабирование.

## 3. Цель продукта

Дать врачу-рентгенологу/радиологу интеллектуального ассистента для рутинных задач: подготовки заключений, поиска протоколов исследований, справочной информации по лучевой диагностике. Продукт должен сократить время на поиск информации и повысить качество заключений за счёт автоматической верификации фактов.

### 3.1 Критерии достижения цели

- [ ] Мария может задать вопрос по протоколу КТ/МРТ/ПЭТ-КТ и получить развёрнутый ответ с **дословными цитатами** из клинических рекомендаций. Цитаты сопровождаются указанием раздела, пункта, версии протокола и даты утверждения
- [ ] medcheck-agent верифицирует факты и ссылки перед выдачей (полный цикл: вопрос → ответ → верификация → выдача)
- [ ] Мария получает ответ ассистента в Telegram, без отображения служебной информации о работе агентов
- [ ] Ассистент корректно разграничивает: помогает с информацией, но не подменяет врача в принятии клинических решений
- [ ] Время получения ответа на типовой запрос — не более 60 секунд. Если ответ занимает более 30 секунд — medhelp-agent отправляет промежуточное сообщение: «Идёт поиск по протоколам и верификация фактов…»
- [ ] Доля ошибочных ответов — не более 10% на этапе MVP. **Ошибка** — фактическое несоответствие ответа обязательным источникам (Минздрав, РОРР). Расхождение между источниками (🔄) ошибкой не считается. Метрика рассчитывается на выборке не менее 50 запросов
- [ ] После 20 запросов Мария подтверждает, что ассистент полезен в рутинной работе (устная или письменная обратная связь)
- [ ] Заключения, составленные с помощью ассистента, корректны по терминологии и ссылкам (оценка Марией)

## 4. Архитектурный подход (заимствовано у Катрин)

### 4.1 Изоляция пользователя

- Каждому специалисту — **свой изолированный агент**.
- Сообщения от Марии не должны попадать в main-агент и не должны быть видны другим пользователям.
- Реализуется через **binding** в конфиге OpenClaw: `peer → agent`.

### 4.2 Двухагентная схема

Используется связка из двух агентов, аналогично katrin-agent + auditor-agent:

| Агент | Роль |
|-------|------|
| **medhelp-agent** (фронтальный) | Общается с Марией в Telegram. Консультирует, помогает составлять заключения, отвечает на вопросы по лучевой диагностике |
| **medcheck-agent** (верификатор) | Проверяет факты, протоколы, ссылки на клинические рекомендации. Работает только через sessions_send от medhelp-agent |

### 4.3 Протокол взаимодействия

```
Мария (Telegram)
    ↓
medhelp-agent → готовит ответ/заключение
    ↓
medhelp-agent → sessions_send → medcheck-agent (факты, ссылки, протоколы)
    ↓
medcheck-agent → проверяет по источникам
    ↓
medcheck-agent → возвращает вердикт
    ↓
medhelp-agent → выдаёт ответ Марии + включает результат проверки
```

### 4.4 Контракт между агентами

**Запрос (medhelp-agent → medcheck-agent):**
```json
{
  "claims": ["строка с фактом для проверки", "..."],
  "sources": ["cr.minzdrav.gov.ru", "rorr.ru"]
}
```
`sources` опционально — если не указан, medcheck-agent выбирает по приоритету.

**Ответ (medcheck-agent → medhelp-agent):**
```json
{
  "verdict": "✅|❌|🔄|❓",
  "details": "текстовое обоснование",
  "verified_claims": ["подтверждённый факт", "..."],
  "sources_used": ["ссылка1", "ссылка2"]
}
```

### 4.5 Обработка ошибок

- Если medcheck-agent не отвечает в течение 45 секунд — medhelp-agent отправляет пользователю: «Проверка источников занимает больше времени, попробую позже. Ваш запрос сохранён.»
- Если источник недоступен — вердикт ❓ с пояснением.
- При недоступности обоих обязательных источников — ответ не выдаётся, medhelp-agent сообщает о временной недоступности службы верификации.

### 4.6 Логирование

- Фиксировать в отдельном хранилище (логовый файл в workspace): ID пользователя, время запроса, отправленные факты, вердикт, итоговый ответ пользователю.
- Хранить минимум 30 дней (для аудита).
- medhelp-agent ведёт лог; medcheck-agent передаёт свой вердикт в составе ответа.

### 4.7 Кеш верификации

- Кеш размещается внутри medcheck-agent (in-memory с периодической записью на диск).
- Срок жизни: 2 недели для источников Минздрава, 1 год для Radiopaedia.
- Инвалидация по дате обновления источников (проверяется при каждом запросе: если дата утверждения протокола новее кеша — факт перепроверяется).
- Формат записи кеша: факт + дата последней верификации + список источников + вердикт.

### 4.8 Принципы (наследуются из katrin-agent)

- **Помощник, а не замена** — ассистент обсуждает любые вопросы в рамках профессиональной деятельности врача-рентгенолога/радиолога: протоколы, анатомию, дифференциальный ряд, клинический контекст исследований, смежные специальности. **Не делает:** не пишет заключения от своего имени, не ставит окончательные диагнозы, не назначает лечение. Все клинические решения остаются за врачом
- **Обязательная верификация** — любой ответ, содержащий факты, ссылки на клинические рекомендации, даты — проверяется medcheck-agent
- **medcheck-agent не генерирует** — только критика и верификация, без предложений
- **Изоляция** — medcheck-agent отвечает только medhelp-agent, другим агентам отказывает

## 6. Источники знаний

Иерархия источников построена по авторитетности для клинической радиологии РФ.

### 6.1 Обязательные источники (приоритет 1 — для верификации клинических фактов)

| Источник | Доступ | Назначение | Примечание |
|----------|--------|------------|------------|
| Клинические рекомендации Минздрава РФ (cr.minzdrav.gov.ru) | web_search + web_fetch | Утверждённые протоколы диагностики и лечения | Официальный нормативный документ. Верификация только по ним — они имеют силу закона |
| Методические рекомендации РОРР (rorr.ru → раздел «Клинические рекомендации») | web_search + web_fetch (PDF) | Практические алгоритмы лучевой диагностики, протоколы сканирования, дозиметрия | Разработаны ведущими радиологами, учитывают отечественный опыт и технику |

Эти два источника — обязательная пара для проверки любого клинического утверждения в области лучевой диагностики в РФ. Если они противоречат друг другу — приоритет у Минздрава, но расхождение фиксируется в вердикте.

### 6.2 Рекомендованные источники (приоритет 2 — для уточнений, терминологии, паттернов)

| Источник | Доступ | Назначение | Примечание |
|----------|--------|------------|------------|
| Radiopaedia.org | web_search + web_fetch | Анатомия, семиотика, дифференциальная диагностика, типичные паттерны | Международный бесплатный энциклопедический ресурс, признанный эталоном для визуализации. Обновляется ежедневно |
| ACR Appropriateness Criteria (acr.org) | web_fetch | Обоснованность назначения лучевых исследований, показания и противопоказания | Американский колледж радиологии — мировой стандарт для выбора оптимального метода. Применим к российской практике, но требует адаптации |
| НРБ-99/2009 (СанПиН) | web_search | Допустимые дозовые нагрузки при лучевых исследованиях | Справочный источник для вопросов по дозиметрии: допустимые дозы, нормы радиационной безопасности |

Эти источники дают глубину — помогают при нестандартных случаях, когда российские протоколы не покрывают детали.

### 6.3 Вспомогательные источники (приоритет 3 — для научного поиска и свежих данных)

| Источник | Доступ | Назначение | Примечание |
|----------|--------|------------|------------|
| PubMed / Google Scholar | web_search | Мета-анализы, систематические обзоры, клинические исследования высокого уровня доказательности | Использовать только если: а) вопрос не решён в обязательных источниках, б) требуется уточнить новые методы (AI, томосинтез, ПЭТ-трассеры). Ограничивать поиск последними 5 годами |
| Вестник рентгенологии и радиологии (vestnik-radiologii.ru) | web_search | Отечественные оригинальные исследования, казуистика | Российский рецензируемый журнал, отражает локальную практику |

### 6.4 Правила верификации

1. **Обязательное правило:** каждый клинический факт (показание, протокол, доза, норма) проверяется по двум источникам из группы «Обязательные». Если там нет данных — подключаются «Рекомендованные».
2. **Приоритет:** Минздрав > РОРР > Radiopaedia > ACR > PubMed.
3. **Расхождения:**
   - Если Минздрав и РОРР расходятся — в вердикте указывается «Требуется уточнение» (🔄) с приведением обеих позиций. Мария принимает решение.
   - Если нет данных в российских источниках, но есть в Radiopaedia/ACR — вердикт «Проверено по международным источникам, нет отечественного утверждённого стандарта» (❓).
4. **Актуальность:** все источники проверяются на дату публикации. Протоколы Минздрава старше 3 лет помечаются как «Устарело» (требуется перепроверка через РОРР или PubMed).

### 6.5 Технические рекомендации для реализации

- Для cr.minzdrav.gov.ru: использовать web_search с site:cr.minzdrav.gov.ru + web_fetch для извлечения текста протокола. Парсить раздел «Дата утверждения».
- Для РОРР: PDF-документы можно извлекать через внешний парсер (pdfplumber). На первое время — использовать web_fetch только для HTML-страниц с аннотациями.
- Для Radiopaedia: прямой web_fetch статьи по URL (известная структура: radiopaedia.org/articles/...). Даёт структурированную информацию по паттернам.
- Для ACR: статичные HTML-страницы, легко парсятся.
- **Кеш:** хранить верифицированные факты с меткой времени (срок жизни: 2 недели для Минздрава, 1 год для Radiopaedia). Детальная схема — п. 4.7.
- Для РОРР: если PDF недоступен — использовать аннотацию с сайта (если есть) или переходить к Radiopaedia с пометкой ❓.
- Для PubMed: фильтр `("meta-analysis"[Publication Type] OR "systematic review"[Publication Type])` — отсекать единичные исследования, использовать только обзоры высокого уровня доказательности.

## 8. Skills и накопление опыта

Ассистент не должен каждый раз начинать с чистого листа — он накапливает опыт в процессе работы с Марией.

### 8.1 Долговременная память (medhelp-agent)

- medhelp-agent ведёт MEMORY.yaml (в папке memory/) со структурой:
  - `preferences` — стиль ответов, частота проверок
  - `frequent_queries` — текст запроса → ссылка на последний ответ
  - `favorite_protocols` — список URL и дат
- При повторяющихся запросах ассистент использует накопленный контекст, а не начинает поиск с нуля
- MEMORY.yaml пополняется автоматически: при каждом диалоге агент дописывает новую запись в конец. Для частых запросов — обновляет существующую запись (а не дублирует)

### 8.2 Кеш верификации (medcheck-agent)

- medcheck-agent ведёт собственную базу проверенных фактов (локально в knowledge-файлах)
- Один раз проверил протокол — запомнил результат. При повторном запросе не перепроверяет, если источник не обновился
- Сроки кеширования (см. п. 6.5): Минздрав — 2 недели, Radiopaedia — 1 год, ACR / PubMed / НРБ-99/2009 — 1 неделя
- **Форсированная перепроверка:** по явному запросу Марии «обнови данные по протоколу X» — кеш для этого факта очищается и данные перепроверяются
- При расхождении с новыми данными — кеш инвалидируется и факт перепроверяется

### 8.3 Skills (Skill Workshop)

Когда появляется устойчивый типовой сценарий, он оформляется как **Skill**:

- Skills хранятся в `memory/SKILLS.yaml`
- Формат Skill:
  ```yaml
  id: "kt_ogk_template"
  trigger: "шаблон КТ ОГК" или "заключение КТ грудной клетки"
  algorithm:
    - запросить клинические данные у Марии (доза, наличие контраста)
    - сгенерировать структуру: "Техника → Находки → Заключение"
    - вставить ссылки на протокол РОРР
  sources:
    - "РОРР КТ ОГК 2024"
    - "Radiopaedia КТ анатомия"
  version: 1.0
  ```
- **Создание:** после 3-го повторения одного сценария агент предлагает Марии оформить Skill. Мария подтверждает.
- Skill создаётся через `skill_workshop` и подключается к medhelp-agent как плагин компетенции

### 8.4 Эволюция знаний

- **Пополнение источников:** по запросу Марии («добавь сайт X») или если агент в ходе web_search находит авторитетный ресурс (домен .gov, .edu, известное медицинское общество) — предлагает добавить, Мария подтверждает
- **Отслеживание обновлений:** раз в сутки (в 03:00) проверять дату на главных страницах cr.minzdrav.gov.ru и rorr.ru. Если дата изменилась — инвалидировать кеш по этому источнику
- Skill-библиотека растёт вместе с опытом использования продукта

### 8.5 Обратная связь

- Если Мария **явно исправляет ответ** агента (например: «неправильно, тут нужно по протоколу 2025 года»), агент:
  1. Фиксирует исправление в MEMORY.yaml (раздел corrections)
  2. Предлагает создать новый или обновить существующий Skill
- Обратная связь не считается ошибкой в метрике п. 3.1 (там учитываются только неподтверждённые факты, а не корректировки предпочтений)

## 9. Этапы реализации

### План работ с примерными сроками (человеко-дни)

| № | Этап | Подзадачи | Дни |
|---|------|-----------|-----|
| 1 | **Архитектура и конфигурация** | Создать структуру агентов (папки, файлы). Написать AGENTS.md, SOUL.md, TOOLS.md для каждого агента. Прописать binding, agentToAgent, модель, переменные окружения | 1 |
| 2 | **Настройка инфраструктуры** | Установить зависимости (pdfplumber, веб-драйверы). Настроить доступ к источникам (cr.minzdrav.gov.ru, rorr.ru). Проверить работу web_search и web_fetch с доверенными источниками | 1 |
| 3 | **Реализация памяти и кеша** | Создать файлы MEMORY.yaml, cache.yaml, SKILLS.yaml в папке memory/. Написать логику чтения/записи, инвалидации кеша | 1 |
| 4 | **Интеграция medcheck-agent** | Настроить sessions_send между агентами. Написать контракт запрос/ответ (JSON). Проверить изоляцию (medcheck-agent не доступен напрямую Марии) | 0.5 |
| 5 | **Наполнение базы знаний (первичное)** | Загрузить ссылки на актуальные протоколы Минздрава и РОРР. Проиндексировать ключевые термины для быстрого поиска (вручную выбрать 5–10 типовых запросов) | 0.5 |
| 6 | **Техническое тестирование** | Рестарт Gateway. Проверить: binding работает, ответы приходят, medcheck-agent вызывается, вердикты отображаются корректно, таймауты соблюдаются | 0.5 |
| 7 | **Пользовательское тестирование (с Марией)** | 10–20 реальных запросов от Марии. Сбор обратной связи. Фиксация ошибок и расхождений | 1 (календарно 1–2 дня с перерывами) |
| 8 | **Доработки по обратной связи** | Исправить критические ошибки, уточнить источники, добавить недостающие протоколы, улучшить формулировки | 1 |
| 9 | **Финальное тестирование** | Повторный прогон запросов, которые вызвали проблемы. Подтверждение Марией работоспособности | 0.5 |
| 10 | **Документация и инструкция** | Написать краткую инструкцию для Марии (как задавать вопросы, как читать вердикты, что делать при ❓ и 🔄). Зафиксировать версию продукта | 0.5 |

**Итого:** ~7.5 рабочих дней (при одном разработчике).

### Критические зависимости

- 🔗 Этап 5 (наполнение базы) → требует завершения этапа 2 (доступ к источникам)
- 🔗 Этап 7 (тестирование с Марией) → требует завершения этапа 6 (техническое тестирование)
- 🔗 Этап 9 (финальное тестирование) → требует завершения этапа 8 (доработки)

## 10. Границы продукта (что НЕ входит)

- Ведение электронной медицинской карты (ЭМК)
- Интеграция с PACS/RIS-системами
- Автоматическое описание снимков (AI-анализ изображений)
- Самостоятельное формирование и подписание заключений (ассистент помогает материалом, но финальный документ готовит и подписывает врач)
- Назначение лечения, выписка рецептов
- Постановка окончательного клинического диагноза

Всё остальное, что помогает врачу в работе — обсуждается без ограничений.
