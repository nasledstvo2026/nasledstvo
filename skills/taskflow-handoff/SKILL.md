# taskflow-handoff

Handoff-протокол для передачи контекста между агентами в multi-agent системе OpenClaw.

## Когда использовать

- Один агент завершил работу и нужно передать наработки другому (например, Катя → Катрин)
- Isolated cron-сессия завершилась, но контекст нужен для следующего запуска
- Нужно зафиксировать состояние задачи, чтобы новый агент не начинал с нуля
- Ручная передача: Кирилл просит «передай Лене, что я нашёл по РЖД»

## Формат handoff

Один fenced code block, который можно скопировать/передать в sessions_send:

```
# HANDOFF: <краткое название работы>
Дата: YYYY-MM-DD · От: <агент/человек> → Кому: <агент/человек>

## 1. Цель
<Что пытаемся сделать. 1-3 предложения. Северная звезда.>

## 2. Контекст
<Почему это делается, для кого, какие ограничения.>

## 3. Текущее состояние
- ✅ ГОТОВО: <что сделано>
- 🔄 В РАБОТЕ: <что частично>
- ⬜ НЕ НАЧАТО: <что не начато>

## 4. Ключевые решения (и почему)
<Какие решения приняты и почему. Самый ценный раздел.>

## 5. Западни и тупики
<Что уже попробовали и не сработало. Что не надо повторять.>

## 6. Релевантные файлы
<Пути к файлам, строкам, отчётам — что конкретно там лежит.>

## 7. Открытые вопросы
<Что осталось, с зависимостями и порядком.>
```

## Процедура

1. Прочитать AGENTS.md и TOOLS.md получателя — понять его возможности и контекст
2. Если есть файл прошлого handoff — прочитать и обновить, не начинать с нуля
3. Заполнить все разделы шаблона
4. Отправить через sessions_send агенту-получателю
5. Если из чата Кирилла — просто выдать кодблоком в ответ

## Правила

- **Состояние, не инструкции.** Что *фактически верно*, не что *надо делать*
- **Ссылайся, не дублируй.** Не копипастить содержимое отчётов — указать путь
- **Западни — обязательно.** Сэкономят следующему агенту часы
- **Никаких секретов.** API-ключи, токены — только «см. ~/.env»
- **Без воды.** Каждая строка должна быть тем, что агент не найдёт в коде сам
