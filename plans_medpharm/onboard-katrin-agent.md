## 1. Целевая архитектура

```
[Telegram: Катрин] → [OpenClaw Gateway через кастомный channel]
                             ↓
                    session:katrin (persistent-агент)
                        ├─ Скилл: katrin-tender-expert
                        ├─ Модель: deepseek-chat
                        ├─ Роль: Генерация → Исправление документов
                        ├─ Своя память: memory/katrin/
                        ├─ Свои знания: knowledge/katrin/ (уже готовы)
                        ├─ Инструменты: read, edit, write, web_search, web_fetch, exec
                        ├─ Cron: пн/ср/пт 09:30 — обновление LegalMCP
                        └─ Связь: sessions_send → auditor-agent (для аудита)
                             ↓
                    session:auditor (persistent-агент)
                        ├─ Скилл: compliance-audit (предстоит создать)
                        ├─ Модель: deepseek-chat
                        ├─ Роль: Только Критика/Audit документов
                        ├─ Своя память: memory/auditor/
                        ├─ Свои знания: knowledge/auditor/ (УФАС, ЕИС, ЖНВЛП)
                        ├─ Инструменты: read, edit, web_search, web_fetch, exec
                        └─ Источники: решения УФАС, реестр контрактов ЕИС, цены ЖНВЛП
```

### Цикл генерации и аудита документов

```
Катрин (запрос на документ)
     ↓
katrin-agent (Генерация черновика)
     ↓ ─── sessions_send → auditor-agent
юрисконсульт-аудитор (Критика — нарушения, риски, несоответствия)
     ↓ ─── ответ → katrin-agent
katrin-agent (Исправление с учётом критики)
     ↓
katrin-agent (Готовый .docx → Катрин)
```

- Base case: один проход (генерация → критика → исправление → отдача)
- Опционально: повторная критика и доработка по запросу Катрин

## 2. Бизнес-требования (ЧТО нужно бизнесу)

1. **Автоматическая генерация документов** по 44-ФЗ/224-ФЗ (допсоглашения, закупки ЛП/МИ)
2. **Юридическая корректность** — документы должны проходить проверку на соответствие:
   - Нормативной базе (44-ФЗ, 224-ФЗ, ПП №620, ПП №1875)
   - Актуальной практике контролирующих органов
   - Реестрам предельных цен на ЖНВЛП
3. **Формат** — готовый .docx для Microsoft Office
4. **Время отклика** — разумное, без многочасовых простоев
5. **Мониторинг** — Лунт (Кирилл) видит статус системы, не вникая в детали

## 3. Системные требования (КАК реализовано)

### 3.1. Многоагентная архитектура

| Компонент | Роль | Ответственность |
|-----------|------|-----------------|
| **katrin-agent** (session:katrin) | Генератор | Создаёт черновик документа на основе контекста. Получает критику → исправляет → отдаёт финал |
| **юрисконсульт-аудитор** (session:auditor) | Критик | Только аудит. Не генерирует, не исправляет. Анализирует переданный документ на compliance |
| **katrin-agent** | Координатор | Оркестрирует цикл: инициирует генерацию, отправляет на аудит, применяет правки, отдаёт результат |

### 3.2. Механизм взаимодействия агентов

- **Межсессионная связь:** `sessions_send(sessionKey="session:auditor", message=...)` из katrin-agent
- **Протокол:** katrin-agent формирует запрос на аудит (черновик + какие риски проверить) → auditor-agent отвечает списком замечаний и рекомендаций
- **Синхронизация:** последовательная (katrin-agent ждёт ответа от auditor-agent через `timeoutSeconds` в sessions_send)
- **Fallback:** если auditor-agent не отвечает — katrin-agent выдаёт документ с пометкой «без аудита»

### 3.3. Источники данных для аудита

| Источник | Назначение | Тип доступа | Статус |
|----------|-----------|-------------|--------|
| **LegalMCP** | Основная нормативная база (44-ФЗ, 224-ФЗ, ПП) | MCP API | 🟢 Работает (84/100 запр.) |
| **pravo.gov.ru** | Тексты НПА | web_fetch | 🟢 Доступен |
| **Решения УФАС** (fas.gov.ru) | Проверка: не сочтут ли допсоглашение ограничением конкуренции | API / парсер | 🔴 Исследовать |
| **Реестр контрактов ЕИС** (zakupki.gov.ru) | Сверка с формами, прошедшими финконтроль | API / парсер | 🔴 Исследовать |
| **Единый реестр предельных цен на ЖНВЛП** | Контроль цен и наименований ЛП | API / выгрузка | 🔴 Исследовать |

### 3.4. Цикл аудита (детально)

```
1. katrin-agent получает запрос от Катрин
2. katrin-agent генерирует документ (python-docx → черновик)
3. katrin-agent отправляет auditor-agent:
   - Текст документа
   - Тип проверки (допсоглашение / закупка / изменение цены / медизделие)
   - Специфические риски по контексту
4. auditor-agent выполняет проверку:
   a. Сверка с LegalMCP / pravo.gov.ru — соответствует ли нормам
   b. УФАС: есть ли решения по аналогичным кейсам
   c. ЕИС: типовые формы уже проходили контроль?
   d. ЖНВЛП: не превышена ли предельная цена
   e. Формирует отчёт: [Нарушение / Риск / OK] по каждому пункту
5. katrin-agent исправляет документ с учётом критики
6. katrin-agent формирует финальный .docx
7. katrin-agent отправляет Катрин + прикладывает короткий комментарий по правкам
```

### 3.5. Разделение окружений

| Аспект | katrin-agent | auditor-agent |
|--------|--------------|---------------|
| Сессия | session:katrin | session:auditor |
| Память | memory/katrin/ | memory/auditor/ |
| Знания | knowledge/katrin/ (законы, процедуры, документы-образцы) | knowledge/auditor/ (УФАС-практика, реестры, комплаенс-правила) |
| AGENTS.md | Тендерный эксперт, генератор документов | Юрисконсульт-аудитор, compliance-специалист |
| Инструменты | read, edit, write, web_search, web_fetch, exec, sessions_send | read, edit, web_search, web_fetch, exec |

## 6. План отката

| Ситуация | Действие |
|----------|----------|
| Катрин не получает ответы | Переключить маршрутизацию обратно на main. Восстановить старый cron |
| LegalMCP-крон падает | Откат крона на main, разбираться отдельно |
| Агенты не синхронизируются | Сделать katrin-agent самодостаточным (аудит опционально) |
| auditor-agent не отвечает | katrin-agent- fallback "без аудита" |
| Полный провал | Отключить session:katrin и session:auditor. Удалить кастомный маршрут. Вернуть всё как было |
