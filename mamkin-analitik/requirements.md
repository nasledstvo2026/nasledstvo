## 1. Executive Summary

**«Создать БТ»** — сервис для формирования бизнес-требований (БТ) высокого уровня зрелости.
Сервис решает проблему низкой культуры работы с БТ: владельцы продуктов и бизнес-аналитики
тратят много времени, используют разные шаблоны, формулируют поверхностные требования без
анализа корневых причин. «Создать БТ» через динамический опрос и мультиагентный контроль
помогает создавать глубокие, структурированные и измеримые бизнес-требования.

## 2. Core Vision

### 2.1. Problem Statement

В организации сложилась неэффективная практика работы с бизнес-требованиями.
Владельцы продуктов (Катя, Лена, Данил, Роман, Кирилл) и бизнес-аналитики:

- Тратят много времени на подготовку БТ
- Используют разные шаблоны — нет единого формата
- Часто подходят формально, не прорабатывают глубину
- Не формулируют проблемную область, не выходят на корневые причины
- Подготовка БТ либо откладывается, либо делается наспех
- Оценка трудозатрат делается «из головы», знания не фиксируются

**Корневая причина:** низкая культура работы с БТ — пользователи не понимают ценность
качественного документа, пренебрегают этим этапом.

### 2.2. Why Existing Solutions Fall Short

- Традиционные шаблоны БТ не адаптируются под ответы пользователя
- Нет механизма «принуждения к глубине» — если пользователь отвечает поверхностно, система не задаёт уточняющих вопросов
- Нет контроля качества: отсутствует ревьюер, который проверяет полноту разделов
- Нет механизма обучения: каждый новый БТ создаётся с нуля без учёта предыдущего опыта
- Нет интеграции с публикацией (DOCX + веб-версия)

### 2.3. Proposed Solution

«Создать БТ» — мультиагентный сервис в Telegram, который:

1. Проводит **динамический опрос** пользователя, адаптируя вопросы под его ответы
2. Использует **мультиагентный пайплайн**:
   - Агент-опросчик задаёт вопросы, погружается в контекст
   - Агент-составитель формирует текст БТ по шаблону
   - Агент-контролёр проверяет полноту и глубину
   - Агент-генератор создаёт DOCX и веб-версию
3. **Самообучается**: с каждым новым БТ качество опроса повышается
4. Генерирует **единый формат** БТ (утверждённый шаблон из 7 блоков)
5. Публикует результат в **DOCX и на GitHub Pages**
6. Интегрируется через **Telegram** (основной канал взаимодействия)

### 2.4. Key Differentiators

- **Динамическая глубина**: при поверхностном ответе агент копает глубже, а не принимает «как есть»
- **Контроль качества в реальном времени**: контролёр проверяет каждый раздел на полноту
- **Самообучаемость**: сервис накапливает опыт от сессии к сессии
- **Мультиагентная архитектура** на базе OpenClaw + DeepSeek API
- **Двойной формат выдачи**: DOCX (для Jira) + веб-страница (для просмотра)
- **Жёсткий шаблон**: 7 фиксированных блоков без лишней художественности

## 4. Описание цели

### Главная цель
Разработать сервис, позволяющий формировать бизнес-требования высокого уровня зрелости.

### Что такое высокий уровень зрелости БТ
Это глубина проработки каждого раздела шаблона. БТ считается зрелой, когда проблемная область проанализирована до корневой причины, цель измерима, заинтересованные стороны идентифицированы, требования конкретны, риски осознаны.

### За счёт чего достигается
1. **Динамический опрос** — агент реагирует на ответы: при поверхностном ответе задаёт уточняющие вопросы («А что стоит за этой болью?», «Кто именно страдает?», «Что будет, если ничего не менять?»), при глубоком — фиксирует и идёт дальше
2. **Мультиагентный контроль** — контролёр проверяет каждый раздел на полноту и возвращает на доработку, если раздел заполнен поверхностно
3. **Самообучаемость** — с каждым новым БТ агент учится задавать более точные вопросы на основе предыдущего опыта

### Business Objectives

1. **Повышение зрелости БТ:** каждый раздел документа соответствует критериям зрелости (см. Раздел 5)
2. **Сокращение времени подготовки БТ** по сравнению с текущим процессом
3. **Документированная оценка трудозатрат:** оценка становится воспроизводимой, а не «знанием в голове» при уходе человека
4. **Повышение культуры работы с БТ:** каждая новая инициатива в Jira содержит БТ, подготовленный сервисом

## 5. Критерии достижения цели (Success Metrics)

### Критерии зрелости БТ (целевые)
Сервис достиг цели, если подготовленный им документ БТ соответствует следующим критериям:
- **Проблемная область** — проанализирована до корневой причины, а не просто описана поверхностно
- **Бизнес-цель** — сформулирована измеримо, привязана к показателям бизнеса
- **Критерии достижения цели** — явно указаны, достижимы и проверяемы
- **Заинтересованные стороны** — идентифицированы все, чьи интересы затронуты
- **Функциональные возможности** — конкретны, без общих фраз
- **Качественные характеристики** — учтены (безопасность, производительность и т.д.)
- **Ограничения** — осознаны и зафиксированы
- **Критерии приёмки** — проверяемы
- **Риски** — осознаны, а не отписка

### Key Performance Indicators

| Метрика | Цель | Измерение |
|---------|------|-----------|
| Полнота разделов БТ | 7/7 блоков заполнены | Автоматическая проверка контролёром |
| Глубина проработки | Экспертная оценка «хорошо» и выше | Качественная оценка (не автоматизирована) |
| Время подготовки БТ | Меньше, чем ручной процесс | Сравнение с текущими трудозатратами |
| Доля инициатив с БТ | 100% новых инициатив | Jira-задачи с прикреплённым БТ |
| Удовлетворённость пользователя | Легко и понятно проходить опрос | Обратная связь от PO и БА |
| Улучшение от сессии к сессии | Качество растёт | Анализ паттернов вопросов (самообучение) |

### Признаки успеха внедрения
- Каждая новая инициатива в Jira содержит БТ, подготовленный сервисом
- На основании БТ проводится экспресс-оценка трудозатрат — оценка становится документированной и воспроизводимой, а не «знанием в голове»
- Время подготовки БТ сократилось по сравнению с текущим процессом
- Пользователю легко и понятно проходить опрос

### Конечный результат работы сервиса
- Готовый документ БТ в формате DOCX
- Публикация идентичной версии на веб-странице на GitHub Pages
- Пользователь самостоятельно прикрепляет DOCX к задаче в Jira (интеграция с Jira не предусмотрена)

## 6. Контекст и ограничения

**Инфраструктура (что есть):**
- VPS (vm-f13581, Linux)
- GitHub Pages для статики
- DeepSeek API (LLM)

**Ограничения сети:**
- Прямой доступ к VPS по HTTP/IP заблокирован в корпоративной сети (но пользователи работают из дома/с мобильного — не критично)

**Язык:** только русский. Мультиязычность не предусмотрена.

**Объём БТ:**
- Зависит от контекста инициативы
- Формулировки без лишней художественности, по делу

## 7. Шаблон БТ (неменяемый) — утверждён

> ⚠️ Шаблон фиксируется на этапе разработки сервиса и не подлежит изменению после ввода в эксплуатацию. Все вопросы агента строятся на основе этой структуры.

### Структура шаблона БТ (агентный формат — 7 диалоговых блоков)

Шаблон организован как последовательность из 7 блоков, каждый из которых содержит подразделы. Агент-опросчик проходит блоки последовательно (Блок 1 → Блок 2 → … → Блок 7), задавая вопросы с динамической глубиной:

- **L1 (открывающий)** — общий вопрос, позволяет пользователю ответить свободно
- **L2 (углубляющий)** — уточняет и конкретизирует, если ответ L1 поверхностный
- **L3 (корневой)** — добирается до коренной причины, если L1-L2 недостаточно

Риски собираются агентом на протяжении всего диалога, фиксируются в память сессии и систематизируются в Блоке 7.

| № | Блок | Подразделы | 
|---|------|------------|
| 1 | **Контекст и проблема** | 1.1 Контекст инициативы · 1.2 Проблемная область · 1.3 Последствия бездействия |
| 2 | **Цель и измерение** | 2.1 Бизнес-цель · 2.2 Критерии успеха |
| 3 | **Стейкхолдеры и влияние** | 3.1 Заинтересованные стороны · 3.2 Их интересы и влияние |
| 4 | **Решение** | 4.1 Ключевые функциональные возможности · 4.2 Пользовательские сценарии · 4.3 Граничные случаи и исключения |
| 5 | **Требования к качеству и ограничения** | 5.1 Качественные характеристики · 5.2 Ограничения |
| 6 | **Приёмка** | 6.1 Критерии приёмки (с привязкой к функциям) |
| 7 | **Риски и митигации** | 7.1 Идентифицированные риски · 7.2 Митигации |

| 6.1 Критерии приёмки | L2 | «Опишите в формате Given-When-Then: Дано: ситуация. Когда: действие. Тогда: ожидаемый результат.» |
| 6.1 Критерии приёмки | L3 | «Кто принимает? Есть ли blocker для приёмки? Нужно ли автоматическое тестирование?» |

#### Блок 7 — Риски и митигации

| Подраздел | Уровень | Вопрос-подсказка |
|-----------|---------|------------------|
| 7.1 Идентифицированные риски | L1 | «Мы по ходу разговора затронули несколько рисков. Давайте их систематизируем. Какие риски вы видите?» |
| 7.1 Идентифицированные риски | L2 | «По категориям: технические, организационные, бизнес-риски, adoption.» |
| 7.1 Идентифицированные риски | L3 | «Для каждого риска: вероятность (низкая/средняя/высокая), влияние (низкое/среднее/критическое), приоритет.» |
| 7.2 Митигации | L1 | «Какие действия можем предпринять, чтобы снизить вероятность или влияние каждого риска?» |
| 7.2 Митигации | L2 | «Для критических рисков: что делаем сейчас для предотвращения? Что будем делать, если наступит? Есть ли план Б?» |
| 7.2 Митигации | L3 | «Есть ли риски, которые можно полностью устранить? Или те, с которыми придётся жить?» |

### Примечания к шаблону:
- Агент последовательно проходит по блокам, задавая вопросы с динамическим уровнем глубины
- L1 → L2 → L3: агент углубляется только при поверхностном ответе; при глубоком ответе — фиксирует и переходит к следующему подразделу
- Риски собираются на протяжении всего диалога (фиксируются в память сессии), систематизируются в Блоке 7
- Блоки заполняются нелинейно: агент может возвращаться к блокам для уточнения
- Все блоки и подразделы обязательны к заполнению (даже ответом «не применимо»)
- Шаблон неменяемый, все блоки фиксированы

## 8. Архитектура решения

### Канал взаимодействия
- Пользователь общается с агентом через **Telegram** (стандартный канал)
- Агент задаёт вопросы, анализирует ответы, динамически адаптирует следующие вопросы

### Принцип работы
- Сервис сам опрашивает пользователя (агентный подход)
- Вопросы динамически меняются в зависимости от ответов
- Агент погружается в суть предмета, адаптирует следующие вопросы

### Мультиагентная архитектура
Архитектурный подход аналогичен тому, что используется для Марии (medhelp-agent) и Катрин (katrin-agent):

```
Telegram → main (OpenClaw) → мультиагентная связка:
  ├─ agent-опросчик    — задаёт вопросы, погружается в контекст
  ├─ agent-составитель  — формирует текст БТ по шаблону
  ├─ agent-контролёр    — проверяет полноту, глубину, отсутствие воды
  └─ agent-генератор    — создаёт DOCX и веб-версию на GitHub Pages
```

### BMAD и business-writing skill в архитектуре сервиса

**BMAD (bmad-method + bmad-master agent)** — «операционная система» для агентов. В сервисе «Создать БТ» BMAD используется как:
1. **Оркестратор мультиагентного пайплайна** — BMAD управляет последовательностью вызова агентов (опросчик → составитель → контролёр → генератор), передаёт контекст между ними, контролирует завершение этапов
2. **Стандартные воркфлоу BMAD** — готовые процессы (Product Brief от Mary, PRD от John) применяются при проектировании новых функций сервиса, а не в продакшн-режиме
3. **Агент Mary (Business Analyst)** — может вызываться как дополнительная экспертиза внутри пайплайна: Mary проверяет сформулированные требования на полноту и даёт рекомендации агенту-контролёру

**Business-writing skill** — даёт агенту навыки профессионального бизнес-аналитика. В сервисе применяется:
1. **Агент-составитель** — использует business-writing для формулирования текста БТ: чёткая структура, обоснованные формулировки, ссылки на контекст
2. **Агент-контролёр** — бизнес-райтинг позволяет точнее оценивать глубину проработки разделов и выявлять поверхностные формулировки
3. **Агент-генератор** — при создании веб-версии и DOCX применяет принципы business-writing для оформления документа

> **Примечание:** BMAD и business-writing — инструменты разработки и внутреннего контроля, а не пользовательские функции. Пользователь взаимодействует только с Telegram-ботом и не видит эти механизмы.

### Риск сервиса (учтён в архитектуре)
- **Основной риск:** агент будет задавать поверхностные вопросы и формулировать высокопарный/водянистый текст
- **Митигация:** multi-agent pipeline с контролёром качества, жёсткий шаблон БТ без лишней художественности

### Самообучаемость
- Сервис должен накапливать опыт от сессии к сессии
- С каждым новым БТ качество опроса должно повышаться
- **Конкретный механизм обучения — уточняется на этапе проектирования**, архитектурно закладывается хранение контекста/паттернов между сессиями (файловая система VPS или БД для кейсов)

### Генерация документа
- Agent (OpenClaw / DeepSeek) формирует БТ на основе утверждённого шаблона
- DOCX-генерация выполняется на VPS (python-docx)

### Публикация
```
[Пользователь (Telegram)] ↔ [мультиагент: опрос → контроль → генерация]
                                                              ↓
                                          DOCX + веб-версия → GitHub Pages
                                                              ↓
                                             Ссылка в Telegram пользователю
```

### Компоненты
- **Канал:** Telegram (бот → пользователь)
- **Agent:** OpenClaw + DeepSeek API — мультиагентная связка
- **Документ:** python-docx (VPS) для генерации .docx
- **Публикация:** GitHub Pages (DOCX + веб-страница)
- **История сессий:** сохраняется для возможности продолжить прерванный опрос и для самообучения сервиса

### Формат финального документа
- **DOCX** — скачивание
- **Веб-версия** — идентичное содержание на GitHub Pages

### Хранение
- История сессий: да (черновики, завершённые БТ, паттерны для самообучения)
- Механизм самообучения: уточняется на этапе проектирования

## 9. MVP Scope

### Core Features (MVP)

1. **Telegram-бот** как основной канал взаимодействия с пользователем
2. **Динамический опрос** по 7 блокам утверждённого шаблона БТ:
   - Агент последовательно проходит по разделам
   - Вопросы адаптируются под ответы пользователя
   - При поверхностном ответе — уточняющие вопросы
3. **Мультиагентный пайплайн:**
   - Агент-опросчик — сбор информации через диалог
   - Агент-составитель — формирование текста по шаблону
   - Агент-контролёр — проверка полноты и глубины разделов
   - Агент-генератор — создание DOCX и веб-версии
4. **Генерация DOCX** (python-docx на VPS)
5. **Публикация на GitHub Pages** (DOCX + веб-страница)
6. **Хранение истории сессий** (черновики, завершённые БТ, паттерны)
7. **Самообучаемость:** накопление опыта для улучшения качества вопросов

### Out of Scope for MVP

- Интеграция с Jira (пользователь прикрепляет DOCX самостоятельно)
- Мультиязычность (только русский язык)
- Агрегация метрик качества (оценка экспертная, не автоматизированная)
- Party Mode и Advanced Elicitation (продвинутые режимы опроса)
- Веб-интерфейс для просмотра сессий (только GitHub Pages для готовых документов)

### MVP Success Criteria

- Сервис успешно проводит опрос по всем 7 блокам
- Контролёр возвращает на доработку поверхностные разделы
- DOCX-файл генерируется корректно
- Веб-версия публикуется на GitHub Pages
- Пользователь получает ссылку в Telegram

### Future Vision (2-3 года)

- Интеграция с Jira для автоматического прикрепления БТ к задачам
- Автоматическая агрегация метрик качества
- Расширение на английский язык
- Веб-интерфейс для управления сессиями
- API для интеграции с другими системами
- Аналитическая панель с метриками по зрелости БТ
