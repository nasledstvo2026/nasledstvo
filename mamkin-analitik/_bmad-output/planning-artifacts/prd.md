# Product Requirements Document — Мамкин аналитик

**Автор:** User
**Дата:** 2026-07-08

## Executive Summary

«Мамкин аналитик» — мультиагентный сервис в Telegram для формирования бизнес-требований (БТ) высокого уровня зрелости. Сервис решает проблему низкой культуры работы с БТ: владельцы продуктов тратят много времени, используют разные шаблоны и формулируют поверхностные требования без анализа корневых причин.

### Что делает продукт особенным

- **Динамическая глубина опроса:** при поверхностном ответе агент задаёт уточняющие вопросы, а не принимает ответ «как есть»
- **Контроль качества в реальном времени:** мультиагентный пайплайн проверяет каждый раздел на полноту и возвращает на доработку
- **Самообучаемость:** сервис накапливает опыт от сессии к сессии, улучшая точность вопросов
- **Мультиагентная архитектура (OpenClaw + DeepSeek API):** агент-опросчик, агент-составитель, агент-контролёр, агент-генератор
- **Двойной формат выдачи:** DOCX (для Jira) + веб-страница (GitHub Pages)
- **Жёсткий утверждённый шаблон:** 7 фиксированных блоков, никакой лишней художественности

### Project Classification

- **Тип проекта:** SaaS B2B (внутренний корпоративный сервис)
- **Домен:** general (инструмент для работы с требованиями)
- **Сложность:** низкая (стандартные требования, без регуляции)
- **Контекст:** brownfield (существует инфраструктура VPS, OpenClaw, DeepSeek API, GitHub Pages)

## Product Scope

### MVP — Minimum Viable Product

1. Telegram-бот как основной канал взаимодействия
2. Динамический опрос по 7 блокам утверждённого шаблона БТ
3. Мультиагентный пайплайн: опросчик → составитель → контролёр → генератор
4. Генерация DOCX (python-docx на VPS)
5. Публикация на GitHub Pages (DOCX + веб-страница)
6. Хранение истории сессий (черновики, завершённые БТ, паттерны)
7. Самообучаемость: накопление опыта для улучшения качества вопросов

### Growth Features (Post-MVP)

- Интеграция с Jira для автоматического прикрепления БТ к задачам
- Автоматическая агрегация метрик качества
- Расширение на английский язык
- Веб-интерфейс для управления сессиями

### Vision (Future)

- API для интеграции с другими системами
- Аналитическая панель с метриками по зрелости БТ
- Party Mode и Advanced Elicitation (продвинутые режимы опроса)

## User Journeys

### Первичный пользователь: Владелец продукта (PO) — успешный сценарий

**Открывающая сцена:** Катя — владелец продукта в компании. У неё новая инициатива, требующая БТ, но времени в обрез, а качественный документ нужен для согласования с командой разработки. Она открывает Telegram и находит бота «Мамкин аналитик» по внутренней ссылке.

**Развитие:** Бот приветствует Катю и объясняет процесс: «Я задам вопросы по 7 разделам. Отвечайте своими словами — я помогу сформулировать глубокие требования». Катя начинает с описания контекста инициативы. Бот слушает, задаёт уточняющие вопросы: «Вы сказали, что процесс отнимает много времени. А что конкретно занимает больше всего?». Катя чувствует, что её действительно понимают.

**Кульминация:** После прохождения всех 7 блоков, бот показывает предварительный результат и спрашивает: «Всё верно? Могу что-то уточнить?». Катя проверяет — все разделы заполнены, формулировки чёткие, без воды. Она нажимает «Завершить».

**Развязка:** Через минуту Катя получает сообщение: «Готово! Вот ваш DOCX и ссылка на веб-версию на GitHub Pages». Она прикрепляет DOCX к задаче в Jira и отправляет команде на согласование. В следующий раз бот задаёт более точные вопросы, учитывая контекст предыдущей сессии.

### Первичный пользователь: Владелец продукта — граничный случай (поверхностные ответы)

**Сцена:** Лена открывает бот, думая «быстро накидать требования». На первый вопрос «Расскажите о проблеме» она отвечает: «Всё плохо, нужно автоматизировать».

**Развитие:** Бот не принимает поверхностный ответ. Контролёр фиксирует: недостаточная глубина. Агент-опросчик переходит на L2: «Что именно плохо? Приведите конкретный пример за последнюю неделю». Лена уточняет. Бот снова проверяет глубину — всё ещё поверхностно. Переходит на L3: «А почему эта ситуация сложилась? Какая коренная причина?».

**Кульминация:** Лена понимает, что «отделаться парой строк» не получится. Бот вежливо, но настойчиво добирается до сути. В итоге Лена формулирует проблему на глубине, о которой даже не задумывалась.

**Развязка:** Лена получает качественный документ и признаёт: «Я бы сама так глубоко не написала». Бот фиксирует: для этой сессии потребовалось больше L3-вопросов, чем в среднем — это данные для самообучения.

### Вторичный пользователь: Разработчик — получатель БТ

**Сцена:** Рома — разработчик. Он открывает задачу в Jira и видит прикреплённый DOCX. Открывает — и сразу видит чёткие функциональные требования без лишней «воды». Каждый пункт конкретен: «Пользователь может указать бюджет проекта», а не «Система должна быть удобной».

**Кульминация:** Рома понимает, что именно нужно реализовать, без необходимости бежать уточнять к PO. Критерии приёмки сформулированы в формате Given-When-Then.

**Развязка:** Рома тратит меньше времени на анализ требований и больше — на реализацию. Качество кода растёт, переделок меньше.

### Третичный пользователь: Руководитель — оценка трудозатрат

**Сцена:** Руководитель видит БТ и может оценить трудозатраты, опираясь на документированные требования, а не на «знание в голове» уходящего сотрудника.

**Кульминация:** Оценка становится воспроизводимой. Если сотрудник уходит, БТ остаётся — и новый человек может оценить объём работы без потери контекста.

## Innovation & Novel Patterns

### Detected Innovation Areas

- **Динамическая глубина опроса:** комбинация адаптивного интервью с multi-level questioning (L1→L2→L3), где система сама решает, когда углубляться, а когда достаточно
- **Мультиагентный пайплайн с контролёром качества:** четыре специализированных агента (опросчик, составитель, контролёр, генератор) работают последовательно, причём контролёр может вернуть на доработку - аналог CI/CD для документов
- **Самообучаемый опросник:** агент учится на предыдущих сессиях, улучшая качество вопросов (patent-pending-подход в контексте БТ)

### Market Context & Competitive Landscape

Традиционные инструменты БТ (Confluence, Notion, Google Docs) предоставляют только статические шаблоны без динамической адаптации. AI-инструменты генерации требований (Jasper, Copy.ai) создают текст, но не проводят структурированный опрос с контролем глубины. «Мамкин аналитик» уникален комбинацией: динамический опрос + мультиагентный контроль + самообучение.

### Validation Approach

- Пилотный запуск на 5 PO внутри компании
- Сравнение времени подготовки БТ с ручным процессом
- Экспертная оценка глубины подготовленных документов
- A/B тестирование: с самообучением vs без (для изолированной оценки эффекта)

### Risk Mitigation

- **Основной риск:** агент формулирует поверхностные/водянистые тексты
  - **Митигация:** жёсткий шаблон БТ, мультиагентный контролёр, итеративные улучшения через самообучение
- **Риск adoption:** PO не хотят тратить время на опрос
  - **Митигация:** преимущество в скорости и качестве vs ручной процесс

## SaaS B2B Specific Requirements

### Project-Type Overview

Внутренний корпоративный сервис для формирования бизнес-требований. Работает как Telegram-бот, оркестрируемый OpenClaw с мультиагентным пайплайном на DeepSeek API.

### Tenant Model

- **Одноарендная модель:** сервис используется внутри одной организации
- **Пользователи разграничиваются на уровне Telegram user ID**
- **Нет мультиарендности в MVP** (все пользователи — сотрудники одной компании)

### Permission Model / RBAC

- **MVP:** базовая модель: пользователь = владелец продукта (все функции доступны)
- **В будущем:** роли «Бизнес-аналитик», «Просмотр» (read-only доступ к истории)
- **Аутентификация:** через Telegram (основной идентификатор), дополнительная авторизация не требуется в MVP

### Subscription / Access Model

- **MVP:** внутренний сервис, доступ по приглашению (ссылке на Telegram-бота)
- **Без биллинга:** сервис внутри компании
- **Авторизация:** по белому списку Telegram user ID

### Integration Requirements

- **GitHub Pages:** публикация DOCX и веб-версии
- **DeepSeek API:** LLM для агентов
- **OpenClaw:** оркестрация мультиагентного пайплайна
- **Telegram Bot API:** канал взаимодействия с пользователем
- **VPS (vm-f13581):** хостинг сервиса
- **Cloudflare Tunnel:** HTTPS-доступ

### Compliance Requirements

- Стандартные требования безопасности для внутреннего сервиса
- Доступ к данным сессий — только у автора и администратора
- Шифрование данных сессий при хранении (рекомендуется)

## Functional Requirements

### Управление сессией

- FR1: Пользователь может начать новую сессию в Telegram-боте
- FR2: Пользователь может продолжить прерванную сессию
- FR3: Пользователь может просмотреть историю своих завершённых сессий
- FR4: Система автоматически сохраняет черновик сессии при каждом ответе пользователя
- FR5: Система завершает сессию по команде пользователя

### Динамический опрос

- FR6: Агент-опросчик последовательно задаёт вопросы по 7 блокам шаблона БТ
- FR7: Агент-опросчик адаптирует уровень глубины вопроса (L1→L2→L3) в зависимости от полноты ответа пользователя
- FR8: При поверхностном ответе агент задаёт уточняющие вопросы, пока не будет достигнута достаточная глубина
- FR9: При полном и глубоком ответе агент фиксирует информацию и переходит к следующему подразделу
- FR10: Система собирает риски на протяжении всего диалога и систематизирует их в Блоке 7
- FR11: Агент может возвращаться к ранее пройденным блокам для уточнения

### Формирование документа

- FR12: Агент-составитель формирует текст БТ по утверждённому шаблону (7 блоков)
- FR13: Агент-составитель формулирует требования чётко, без художественных оборотов
- FR14: Агент-контролёр проверяет каждый раздел на полноту и глубину
- FR15: Если раздел заполнен поверхностно, контролёр возвращает его на доработку агенту-опросчику
- FR16: Агент-контролёр может запросить уточнения у пользователя через агента-опросчика

### Генерация и публикация

- FR17: Агент-генератор создаёт DOCX-файл на VPS (python-docx)
- FR18: Агент-генератор создаёт веб-версию документа
- FR19: Система публикует DOCX и веб-версию на GitHub Pages
- FR20: Система отправляет пользователю ссылку на DOCX и веб-версию в Telegram
- FR21: DOCX и веб-версия идентичны по содержанию

### Самообучение

- FR22: Система сохраняет историю сессий (черновики, завершённые БТ, контекст диалога)
- FR23: Система анализирует паттерны вопросов и ответов для улучшения точности
- FR24: Качество опроса повышается от сессии к сессии за счёт накопленного опыта
- FR25: Система может использовать предыдущие БТ как контекст при формировании новых

### Управление пользователями

- FR26: Система идентифицирует пользователя по Telegram user ID
- FR27: Администратор может добавлять/удалять пользователей в белый список
- FR28: Пользователь может просматривать только свои сессии

## Non-Functional Requirements

### Performance

- NFR1: Ответ агента на сообщение пользователя в Telegram — не более 5 секунд (95-й перцентиль)
- NFR2: Генерация DOCX — не более 30 секунд
- NFR3: Публикация на GitHub Pages — не более 60 секунд
- NFR4: Загрузка истории сессий — не более 3 секунд

### Security

- NFR5: Доступ к сессиям — только у автора и администратора
- NFR6: Данные сессий хранятся с шифрованием (рекомендуется AES-256 для файлов)
- NFR7: DeepSeek API-ключи хранятся в переменных окружения, не в коде
- NFR8: Cloudflare Tunnel обеспечивает HTTPS для всех соединений

### Reliability

- NFR9: Доступность сервиса: 99% в рабочее время (пн–пт, 09:00–19:00 МСК)
- NFR10: Восстановление после сбоя: не более 1 часа
- NFR11: Автоматическое сохранение черновика при каждом ответе (fallback при сбое)

### Scalability

- NFR12: Сервис поддерживает до 20 одновременных сессий
- NFR13: Время ответа не ухудшается при 20 параллельных опросах
- NFR14: Хранение сессий: не менее 500 завершённых БТ

### Accessibility

- NFR15: Telegram-бот поддерживает текстовый ввод (без голосовых команд в MVP)
