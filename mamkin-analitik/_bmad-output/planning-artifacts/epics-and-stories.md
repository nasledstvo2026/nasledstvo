# Мамкин аналитик — Epic Breakdown

## Overview

Этот документ предоставляет полную декомпозицию эпиков и историй для проекта «Мамкин аналитик» — мультиагентного сервиса в Telegram для формирования бизнес-требований высокого уровня зрелости. Документ основан на PRD (35 FR, 15 NFR), Architecture, UX Design Specification и утверждённых требованиях.

## Requirements Inventory

### Functional Requirements

**Управление сессией (FR1–FR5):**
- FR1: Пользователь может начать новую сессию в Telegram-боте
- FR2: Пользователь может продолжить прерванную сессию
- FR3: Пользователь может просмотреть историю своих завершённых сессий
- FR4: Система автоматически сохраняет черновик сессии при каждом ответе пользователя
- FR5: Система завершает сессию по команде пользователя

**Динамический опрос (FR6–FR11):**
- FR6: Агент-опросчик последовательно задаёт вопросы по 7 блокам шаблона БТ
- FR7: Агент-опросчик адаптирует уровень глубины вопроса (L1→L2→L3) в зависимости от полноты ответа пользователя
- FR8: При поверхностном ответе агент задаёт уточняющие вопросы, пока не будет достигнута достаточная глубина
- FR9: При полном и глубоком ответе агент фиксирует информацию и переходит к следующему подразделу
- FR10: Система собирает риски на протяжении всего диалога и систематизирует их в Блоке 7
- FR11: Агент может возвращаться к ранее пройденным блокам для уточнения

**Формирование документа (FR12–FR16):**
- FR12: Агент-составитель формирует текст БТ по утверждённому шаблону (7 блоков)
- FR13: Агент-составитель формулирует требования чётко, без художественных оборотов
- FR14: Агент-контролёр проверяет каждый раздел на полноту и глубину
- FR15: Если раздел заполнен поверхностно, контролёр возвращает его на доработку агенту-опросчику
- FR16: Агент-контролёр может запросить уточнения у пользователя через агента-опросчика

**Генерация и публикация (FR17–FR21):**
- FR17: Система создаёт DOCX-файл с корректным форматированием и стилизацией
- FR18: Агент-генератор создаёт веб-версию документа
- FR19: Система публикует DOCX и веб-версию на GitHub Pages
- FR20: Система отправляет пользователю ссылку на DOCX и веб-версию в Telegram
- FR21: DOCX и веб-версия идентичны по содержанию

**Самообучение (FR22–FR25):**
- FR22: Система сохраняет историю сессий (черновики, завершённые БТ, контекст диалога)
- FR23: Система анализирует паттерны вопросов и ответов для улучшения точности
- FR24: Качество опроса повышается от сессии к сессии за счёт накопленного опыта
- FR25: Система может использовать предыдущие БТ как контекст при формировании новых

**Управление пользователями (FR26–FR28):**
- FR26: Система идентифицирует пользователя по Telegram user ID
- FR27: Администратор может добавлять/удалять пользователей в белый список
- FR28: Пользователь может просматривать только свои сессии

### Additional Requirements (из Architecture и UX)

**Архитектурные требования:**
- Создание структуры проекта по шаблону OpenClaw agent architecture (раздел 5.1 architecture.md)
- Реализация 4 агентов: questioner, compiler, controller, generator
- Файловое JSON-хранилище сессий (без шифрования — снято 09.07.2026)
- Единый Session Context для обмена между агентами
- Шаблон БТ — immutable template.json (7 блоков, L1/L2/L3 вопросы)
- Self-learning через history.json (append-only)
- GitHub Pages публикация через git push
- Cloudflare Tunnel для HTTPS
- Системный шаблон сессий: `/var/mamkin-analitik/sessions/{active,completed}/`
- Retry-логика при сбоях DeepSeek API (3 попытки с exponential backoff)

**UX-требования:**
- Фиксированная структура сообщения бота: Progress → Reflection → Question → Action
- 5 конверсационных компонентов: Progress Tracker, Depth Indicator, Reflection Message, Quality Feedback, Session Resume Card
- Тональность «Коуч / фасилитатор» — тёплый, но профессиональный тон
- L1→L2→L3 с визуальными индикаторами (🔍) и объяснением причины уточнения
- Принцип «терапевта» — без оценок ответов, с нейтральными подтверждениями
- Анти-паттерны: запрет на длинные сообщения, оценочные суждения, техжаргон
- Бесшовное восстановление сессии с показом последнего ответа
- Каждое сообщение ≤ 3 предложений, ≤ 400 символов
- Inline keyboard ≤ 3 кнопок в ряд, ≤ 2 рядов
- 1 эмодзи на сообщение, не ключевой для смысла

## Epic 1: Управление доступом и сессиями пользователей

**And** настроен `.env` с DeepSeek API-ключом, путями к хранилищу и конфигурацией Cloudflare Tunnel

**And** установлены зависимости (python-docx, OpenClaw connectors)

### Story 1.2: Реализация Session Manager (файловое JSON-хранилище)

As a **система**, I want **реализовать Session Manager для сохранения, загрузки и управления сессиями в файловом JSON-хранилище**, так чтобы **каждая сессия сохранялась, восстанавливалась и была изолирована по пользователю**.

**Acceptance Criteria:**

**Given** модуль `lib/session-manager.js` реализован
**When** пользователь отправляет первое сообщение
**Then** Session Manager создаёт новый файл сессии `session_{telegramId}_{sessionId}.json` в `session-storage/active/`
**And** файл содержит: sessionId, telegramUserId, status ("in_progress"), createdAt, updatedAt, template progress, answers, risks
**When** пользователь отправляет ответ
**Then** Session Manager обновляет файл сессии с сохранением всех данных
**And** данные записываются в формате JSON (без шифрования — снято 09.07.2026)
**When** пользователь возвращается после перерыва
**Then** Session Manager загружает активную сессию из `session-storage/active/` и восстанавливает контекст
**When** сессия завершается
**Then** файл перемещается из `session-storage/active/` в `session-storage/completed/`
**And** статус меняется на "completed"
**When** происходит сбой (crash/restart)
**Then** все активные сессии маркируются как "recovered", пользователь уведомляется при следующем сообщении
**And** данные не теряются благодаря транзакционной записи (write to temp → rename)

### Story 1.3: Реализация Template Engine и шаблона БТ

As a **система**, I want **реализовать Template Engine, загружающий immutable template.json с 7 блоками БТ и L1/L2/L3 вопросами**, так чтобы **агенты пайплайна имели единый источник истины для структуры опроса**.

**Acceptance Criteria:**

**Given** файл `data/template.json` создан на основе шаблона из requirements.md (раздел 7)
**When** Template Engine загружается
**Then** он читает `data/template.json` и предоставляет API:
  - `getBlockCount()` → 7
  - `getBlock(номер)` → объект блока с подразделами
  - `getSubsection(block, subsection)` → вопросы L1/L2/L3
  - `getNextSubsection(block, subsection)` → следующий подраздел
**When** Template Engine работает в режиме read-only
**Then** любые попытки записи в template.json игнорируются
**And** структура шаблона соответствует 7 блокам с подразделами, L1/L2/L3 вопросами
**When** агент-опросчик запрашивает вопрос для подраздела 1.1 на уровне L1
**Then** Template Engine возвращает: "Расскажите в двух словах: над чем работаете? Что за проект или инициативу мы обсуждаем?"

### Story 1.4: Интеграция Telegram Bot Connector и whitelist

As a **пользователь**, I want **получить доступ к боту через Telegram по приглашению**, так чтобы **только авторизованные сотрудники могли использовать сервис**.

**Acceptance Criteria:**

**Given** Telegram Bot connector настроен через OpenClaw
**When** пользователь (не в whitelist) отправляет `/start`
**Then** бот отвечает: "Извините, у вас нет доступа к этому сервису. Обратитесь к администратору."

**Then** бот идентифицирует пользователя по Telegram user ID
**And** начинает процесс онбординга
**When** администратор отправляет команду `/adduser <telegram_id>`
**Then** пользователь добавляется в `data/whitelist.json`
**When** администратор отправляет `/removeuser <telegram_id>`
**Then** пользователь удаляется из `data/whitelist.json`
**And** его активные сессии не прерываются, но новые начать нельзя

### Story 1.5: Команды управления сессией (начать, продолжить, просмотр истории, завершить)

As a **владелец продукта**, I want **управлять своими сессиями через простые команды в Telegram**, так чтобы **быстро начать опрос, вернуться к прерванному, посмотреть историю или завершить сессию**.

**Acceptance Criteria:**

**Given** пользователь авторизован
**When** пользователь отправляет `/start`
**Then** бот проверяет наличие активной (незавершённой) сессии
**And** если активная сессия есть — предлагает продолжить: "У вас есть незавершённая сессия от [дата]. Продолжить?"
**And** если активной сессии нет — начинает новую с приветствия
**When** пользователь отправляет `/history`
**Then** бот показывает список завершённых сессий (дата, статус)
**And** предлагает открыть любой завершённый документ
**When** пользователь отправляет `/cancel`
**Then** бот подтверждает: "Вы уверены, что хотите завершить сессию? Черновик будет сохранён."
**And** при подтверждении — сессия завершается, ответы сохраняются
**When** пользователь отправляет `/back`
**Then** бот возвращается к предыдущему подразделу/блоку
**And** показывает последний сохранённый ответ пользователя

As a **система**, I want **автоматически сохранять черновик сессии при каждом ответе пользователя**, так чтобы **ни один ответ не был потерян при сбое или перерыве**.

**Acceptance Criteria:**

**Given** пользователь проходит опрос
**When** пользователь отправляет ответ
**Then** Session Manager сохраняет ответ в файл сессии в `session-storage/active/`
**And** время сохранения не превышает 2 секунд
**And** используется транзакционная запись (временный файл → rename → удаление старого)
**When** происходит сбой во время сохранения
**Then** предыдущая версия файла остаётся неповреждённой
**When** пользователь возвращается после паузы > 30 минут
**Then** Session Resume Card показывает последний сохранённый ответ
**When** файловая система переполнена или недоступна
**Then** пользователь получает уведомление: "⚠️ Не удалось сохранить ответ. Попробуйте позже."

**When** данные передаются между агентами

## Epic 2: Динамический опрос по шаблону БТ

Пользователь проходит адаптивный опрос: агент-опросчик последовательно задаёт вопросы по 7 блокам шаблона, регулирует глубину (L1→L2→L3) в зависимости от полноты ответов, собирает риски и может возвращаться к пройденным блокам. Все сообщения следуют UX-дизайну (тональность «Коуч», фиксированная структура, конверсационные компоненты).

### Story 2.1: Агент-опросчик — базовая логика последовательного опроса

As a **владелец продукта**, I want **получать вопросы от агента-опросчика последовательно по блокам шаблона**, так чтобы **пройти все 7 блоков БТ шаг за шагом**.

**Given** агент-опросчик (`agents/questioner/`) реализован
**When** начинается опрос
**Then** агент-опросчик загружает template.json через Template Engine
**And** начинает с Блока 1, Подраздела 1.1
**And** задаёт L1-вопрос из шаблона
**When** пользователь отвечает на текущий подраздел
**Then** ответ сохраняется в session.answers
**And** агент переходит к следующему подразделу (или к следующему блоку, если подразделы завершены)
**When** все 7 блоков пройдены
**Then** агент-опросчик сигнализирует: "Опрос завершён, передаю данные составителю"

**When** пользователь даёт развёрнутый ответ (≥ 3 предложений или ≥ 2 фактов)
**Then** агент фиксирует ответ на L1-уровне

**Then** ответ сохраняется с `depthReached: "L1"`, и агент переходит к следующему подразделу

### Story 2.3: Агент-опросчик — глубина L2 (уточняющие вопросы)

As a **владелец продукта**, I want **получать уточняющие вопросы при поверхностном ответе**, так чтобы **агент помог мне проработать требование глубже**.

**Acceptance Criteria:**

**Given** пользователь дал поверхностный ответ на L1-вопрос (< 3 предложений или < 2 фактов)
**When** система оценивает глубину
**Then** агент-опросчик переходит на уровень L2
**And** задаёт L2-вопрос из шаблона: "🔍 Давайте уточним: [конкретный вопрос]"
**And** вопрос контекстно-релевантен предыдущему ответу пользователя
**When** пользователь даёт развёрнутый ответ на L2
**Then** ответ сохраняется с `depthReached: "L2"` и агент переходит к следующему подразделу
**When** пользователь снова отвечает поверхностно
**Then** агент переходит на L3 (корневой вопрос)
**Перед L2-вопросом** агент объясняет причину: "Чтобы лучше понять ситуацию, уточню..."

### Story 2.4: Агент-опросчик — глубина L3 (корневые вопросы)

As a **владелец продукта**, I want **получать корневые вопросы, когда даже L2-уточнения недостаточно**, так чтобы **добраться до истинной причины проблемы**.

**When** пользователь отвечает на L3
**Then** ответ сохраняется с `depthReached: "L3"`

**Then** агент фиксирует максимальный достигнутый уровень: `depthReached: "L3"`
**And** переходит к следующему подразделу (без бесконечного цикла)
**Тональность L3-вопросов** — без раздражения, с объяснением: "Чтобы понять коренную причину..."

**When** агент-опросчик выявляет потенциальный риск (по ключевым словам или контексту)

**When** пользователь доходит до Блока 7 (Риски и митигации)
**Then** агент показывает все собранные риски: "По ходу разговора я заметил несколько рисков. Давайте их систематизируем."
**When** пользователь уточняет/дополняет риски
**Then** обновлённые данные сохраняются в сессию
**And** для каждого риска запрашивается: вероятность, влияние, митигация

**Then** он объясняет причину: "Давайте уточним раздел [название]. [причина]"
**And** задаёт конкретный вопрос по недостаточно проработанной части
**When** пользователь завершил уточнение

## Epic 3: Контроль качества и формирование черновика БТ

После завершения опроса агент-составитель формирует структурированный текст БТ по утверждённому шаблону (7 блоков). Агент-контролёр проверяет каждый раздел на полноту и глубину. Если раздел поверхностный — возвращает на доработку агенту-опросчику с конкретными замечаниями.

### Story 3.1: Агент-составитель — формирование текста по шаблону

As a **владелец продукта**, I want **чтобы на основе моих ответов был сформирован структурированный документ БТ по шаблону из 7 блоков**, так чтобы **получить готовый, профессионально оформленный документ**.

**Acceptance Criteria:**

**Given** все 7 блоков опроса завершены
**When** агент-составитель (`agents/compiler/`) получает session.answers
**Then** он формирует черновик `session.draft` в формате:
  - Для каждого из 7 блоков: заголовок блока + заполненные подразделы
  - Формулировки чёткие, без художественных оборотов (FR13)
  - Ответы пользователя переформулированы в формальный стиль БТ
**When** чёрновик сформирован
**Then** каждый раздел соответствует структуре template.json
**And** формулировки конкретны: "Пользователь может указать бюджет проекта", а не "Система должна быть удобной"
**Когда** ответ на подраздел отсутствует или "не применимо"
**Then** раздел помечается как "Не применимо (указано пользователем)"

---

### Story 3.2: Агент-контролёр — проверка полноты разделов

As a **система**, I want **агент-контролёр проверял каждый раздел на полноту**, так чтобы **ни один из 7 блоков не остался незаполненным или поверхностным**.

**Acceptance Criteria:**

**Given** агент-составитель сформировал `session.draft`
**When** агент-контролёр (`agents/controller/`) проверяет полноту
**Then** каждый из 7 блоков проверяется на наличие содержимого
**And** каждый подраздел проверяется на наличие осмысленного текста (не "да", "нет", "не знаю")
**When** блок или подраздел не заполнен
**Then** контролёр фиксирует: `{ block: N, subsection: N.M, issue: "empty", severity: "error" }`
**When** все 7 блоков заполнены
**Then** проверка полноты считается пройденной

---

### Story 3.3: Агент-контролёр — проверка глубины проработки

As a **система**, I want **агент-контролёр проверял глубину проработки каждого раздела**, так чтобы **документ БТ был высокого качества, а не формальной отпиской**.

**Acceptance Criteria:**

**Given** проверка полноты пройдена
**When** агент-контролёр проверяет глубину
**Then** для каждого подраздела оценивается:
  - Содержит ли ответ конкретные факты/цифры (не общие фразы)
  - Есть ли анализ причин (для проблемного блока)
  - Есть ли измеримые показатели (для цели)
  - Идентифицированы ли конкретные стороны (для стейкхолдеров)
  - Конкретны ли функциональные требования (для решения)
**When** раздел оценён как поверхностный
**Then** контролёр фиксирует: `{ block: N, issue: "shallow", reason: "конкретная причина", suggestions: [рекомендации] }`
**When** раздел оценён как достаточный
**Then** раздел маркируется `approved: true`

---

### Story 3.4: Механизм возврата на доработку (контролёр → опросчик → пользователь)

As a **владелец продукта**, I want **получать конкретные рекомендации по доработке, а не просто "недостаточно глубоко"**, так чтобы **понимать, что именно нужно уточнить**.

**Acceptance Criteria:**

**Given** контролёр обнаружил поверхностный раздел
**When** формируется запрос на доработку
**Then** запрос содержит: номер блока, подраздел, конкретную причину ("раздел содержит общие фразы, нет цифр")
**And** рекомендации: "Укажите конкретные KPI: сколько пользователей, какое время отклика"
**When** запрос возвращается агенту-опросчику
**Then** опросчик задаёт уточняющий вопрос в тональности Quality Feedback:
  "💡 Я заметил, что этот раздел можно проработать глубже. Конкретно: [причина]. Давайте уточним: [вопрос]"
**When** после доработки
**Then** цикл повторяется: составитель → контролёр
**When** количество итераций превышает лимит (max 3)
**Then** контролёр форсированно принимает раздел: "ОК, зафиксируем как есть"

---

### Story 3.5: Запрос уточнений у пользователя через агента-опросчика

As a **система**, I want **агент-контролёр мог запросить уточнения через агента-опросчика**, так чтобы **пользователь не взаимодействовал напрямую с контролёром, а получал естественные вопросы от опросчика**.

**Acceptance Criteria:**

**Given** контролёру не хватает информации в ответах
**When** контролёр формирует запрос на уточнение
**Then** запрос обрабатывается агентом-опросчиком
**And** опросчик задаёт вопрос пользователю в естественной форме
**When** пользователь отвечает
**Then** ответ передаётся обратно составителю/контролёру через Session Context
**Пользователь не видит** прямых сообщений от контролёра — только через опросчика

---

### Story 3.6: Формирование финального черновика

As a **система**, I want **после прохождения всех проверок сформировать финальный черновик БТ**, так чтобы **агент-генератор получил готовый текст для создания DOCX и веб-версии**.

**Acceptance Criteria:**

**Given** все проверки контролёра пройдены (approved)
**When** формируется финальный черновик
**Then** `session.draft` содержит 7 блоков с заполненными подразделами
**And** документ содержит секцию рисков (собранных на протяжении диалога, систематизированных в Блоке 7)
**And** формулировки требований конкретны, без художественных оборотов
**And** `session.documentReady` устанавливается в `true`
**And** Session Manager сохраняет финальную версию черновика

## Epic 4: Генерация DOCX, веб-версии и публикация

Агент-генератор создаёт DOCX (через python-docx на VPS) и веб-версию документа, публикует на GitHub Pages и отправляет пользователю ссылки в Telegram. DOCX и веб-версия идентичны по содержанию.

### Story 4.1: Агент-генератор — создание DOCX-файла

As a **владелец продукта**, I want **получить профессионально оформленный DOCX-файл БТ**, так чтобы **прикрепить его к задаче в Jira и отправить команде**.

**Acceptance Criteria:**

**Given** финальный черновик готов (`session.documentReady = true`)
**When** запускается генерация DOCX
**Then** `agents/generator/docx-gen.py` создаёт DOCX-файл с:
  - Титульной страницей: название проекта, дата, автор
  - Оглавлением
  - 7 разделов с заголовками H1/H2/H3
  - Таблицами для критериев приёмки
  - Стилями: заголовки — тёмно-синий (#1A365D), текст — тёмно-серый (#2D3748)
  - Нумерацией страниц
**When** файл создан
**Then** он сохраняется в временную директорию
**And** время генерации ≤ 30 секунд (NFR2)
**When** генерация завершается ошибкой
**Then** retry-механизм (3 попытки с exponential backoff)
**And** при полной неудаче: пользователь получает уведомление "Технический сбой при генерации DOCX"

---

### Story 4.2: Агент-генератор — создание веб-версии

As a **владелец продукта**, I want **получить веб-версию БТ (HTML)**, так чтобы **поделиться ссылкой с коллегами без скачивания файла**.

**Acceptance Criteria:**

**Then** `agents/generator/html-gen.py` создаёт HTML-страницу с:
  - Тем же содержанием, что DOCX (FR21)
  - Современным, читаемым дизайном (шрифты, отступы, цвета согласно UX Design Specification)
  - Адаптивной вёрсткой для мобильных и десктопа
  - Печатью (CSS @media print)
**When** веб-версия создана
**Then** она сохраняется во временную директорию

---

### Story 4.3: Публикация на GitHub Pages

As a **система**, I want **публиковать DOCX и веб-версию на GitHub Pages**, так чтобы **документы были доступны по постоянной ссылке**.

As a **владелец продукта**, I want **получить сообщение со ссылками на DOCX и веб-версию в Telegram**, так чтобы **сразу открыть документ и использовать его**.

**Then** сообщение: "✅ DOCX готов: [ссылка]\n🌐 Веб-версия временно недоступна"

## Epic 5: Самообучение и накопление опыта

Система сохраняет историю сессий, анализирует паттерны вопросов и ответов, использует предыдущий опыт для повышения качества опроса. Self-learning обновляется асинхронно после завершения сессии.

### Story 5.1: Сохранение истории сессий для самообучения

As a **система**, I want **сохранять историю завершённых сессий в history.json**, так чтобы **накапливать данные для анализа и самообучения**.

**Acceptance Criteria:**

**Given** сессия завершена (статус "completed")
**When** система обновляет `history.json`
**Then** в файл добавляется запись с sessionId, telegramUserId, completedAt, durations
**And** агрегированные паттерны: `avgDepthPerBlock`, `frequentL3Topics`, `commonGaps`
**And** `history.json` работает в режиме append-only (не перезаписывается целиком)
**When** загружается история сессий
**Then** время загрузки ≤ 3 секунд (NFR4)
**When** количество сессий достигает 500
**Then** система продолжает штатно работать (NFR14)

---

### Story 5.2: Анализ паттернов вопросов и ответов

As a **система**, I want **анализировать паттерны вопросов и ответов из истории сессий**, так чтобы **выявлять темы, требующие более частых L3-уточнений, и улучшать точность вопросов**.

**Acceptance Criteria:**

**Given** `history.json` содержит данные ≥ 1 завершённой сессии
**When** запускается анализ паттернов (асинхронно после завершения сессии)
**Then** система анализирует:

### Story 5.3: Использование предыдущих БТ как контекста для новых сессий

As a **владелец продукта**, I want **чтобы бот задавал более точные вопросы с учётом предыдущих сессий**, так чтобы **каждый новый опрос был персонализированным**.

**And** приветствие учитывает историю: "С возвращением! В прошлый раз вы работали над [проект X]"
**When** задаются вопросы по разделам, где у пользователя в прошлый раз были проблемы
**Then** агент формулирует вопросы более конкретно и с учётом предыдущего контекста
**When** у пользователя нет истории
**Then** опрос проходит по стандартному шаблону

### Story 5.4: Асинхронное обновление self-learning после завершения сессии

As a **система**, I want **self-learning обновлялся асинхронно после завершения сессии**, так чтобы **не задерживать выдачу документа пользователю**.

**Given** сессия завершена и документ отправлен пользователю

**And** данные не теряются
