# Product Brief: Мамкин аналитик

## Executive Summary

«Мамкин аналитик» — сервис для формирования бизнес-требований (БТ) высокого уровня зрелости. 
Сервис решает проблему низкой культуры работы с БТ: владельцы продуктов и бизнес-аналитики 
тратят много времени, используют разные шаблоны, формулируют поверхностные требования без 
анализа корневых причин. «Мамкин аналитик» через динамический опрос и мультиагентный контроль 
помогает создавать глубокие, структурированные и измеримые бизнес-требования.

---

## Core Vision

### Problem Statement

В организации сложилась неэффективная практика работы с бизнес-требованиями. 
Владельцы продуктов (Катя, Лена, Данил, Роман, Кирилл) и бизнес-аналитики:

- Тратят много времени на подготовку БТ
- Используют разные шаблоны — нет единого формата
- Часто подходят формально, не прорабатывают глубину
- Не формулируют проблемную область, не выходят на корневые причины
- Подготовка БТ либо откладывается, либо делается наспех
- Оценка трудозатрат делается «из головы», знания не фиксируются

**Корневая причина:** низкая культура работы с БТ — пользователи не понимают ценность 
качественного документа, пренебрегают этим этапом.

### Why Existing Solutions Fall Short

- Традиционные шаблоны БТ не адаптируются под ответы пользователя
- Нет механизма «принуждения к глубине» — если пользователь отвечает поверхностно, 
  система не задаёт уточняющих вопросов
- Нет контроля качества: отсутствует ревьюер, который проверяет полноту разделов
- Нет механизма обучения: каждый новый БТ создаётся с нуля без учёта предыдущего опыта
- Нет интеграции с публикацией (DOCX + веб-версия)

### Proposed Solution

«Мамкин аналитик» — мультиагентный сервис в Telegram, который:

1. Проводит **динамический опрос** пользователя, адаптируя вопросы под его ответы
2. Использует **мультиагентный пайплайн**:
   - Агент-опросчик задаёт вопросы, погружается в контекст
   - Агент-составитель формирует текст БТ по шаблону
   - Агент-контролёр проверяет полноту и глубину
   - Агент-генератор создаёт DOCX и веб-версию
3. **Самообучается**: с каждым новым БТ качество опроса повышается
4. Генерирует **единый формат** БТ (утверждённый шаблон из 7 блоков)
5. Публикует результат в **DOCX и на GitHub Pages**
6. Интегрируется через **@Sbernasledtsvo_bot** в Telegram (основной канал взаимодействия, OpenClaw gateway)

### Key Differentiators

- **Динамическая глубина**: при поверхностном ответе агент копает глубже, а не принимает «как есть»
- **Контроль качества в реальном времени**: контролёр проверяет каждый раздел на полноту
- **Самообучаемость**: сервис накапливает опыт от сессии к сессии
- **Мультиагентная архитектура** на базе OpenClaw + DeepSeek API
- **Двойной формат выдачи**: DOCX (для Jira) + веб-страница (для просмотра)
- **Жёсткий шаблон**: 10 фиксированных разделов без лишней художественности

## Target Users

### Primary Users

**Владельцы продуктов (Product Owners)**
- Катя, Лена, Данил, Роман, Кирилл — основные пользователи сервиса
- Инициируют новые продуктовые инициативы
- Отвечают за формирование БТ, но не имеют для этого достаточных инструментов и культуры
- Мотивация: быстро получить качественный документ БТ, снизить нагрузку по подготовке
- Боль: тратят много времени, не уверены в полноте и глубине документа

**Бизнес-аналитики (Business Analysts)**
- Определяются по мере внедрения сервиса
- Профессионально занимаются подготовкой БТ
- Заинтересованы в едином стандарте и глубине проработки
- Боль: нет единого шаблона, формат зависит от конкретного специалиста

### Secondary Users

**Разработчики и команда разработки**
- Косвенные пользователи — получают БТ как вход для разработки
- Заинтересованы в качественных, полных и конкретных требованиях
- Выигрывают от снижения неопределённости и переформулировок

**Руководители / Стейкхолдеры**
- Оценивают трудозатраты на основе БТ
- Заинтересованы в документированной и воспроизводимой оценке, а не «знании в голове»

### User Journey

1. **Discovery:** Пользователь узнаёт о сервисе через Telegram (ссылка/приглашение)
2. **Onboarding:** Бот приветствует, объясняет процесс, запрашивает контекст инициативы
3. **Core Usage:** Динамический опрос — агент задаёт вопросы по 7 блокам шаблона, уточняет, контролирует глубину
4. **Success Moment:** Пользователь получает готовый DOCX + ссылку на GitHub Pages
5. **Long-term:** Пользователь прикрепляет DOCX к задаче в Jira. В следующий раз сервис задаёт более точные вопросы (самообучение)

## Success Metrics

### Business Objectives

1. **Повышение зрелости БТ:** каждый раздел документа соответствует критериям зрелости:
   - Проблемная область проанализирована до корневой причины
   - Бизнес-цель измерима и привязана к показателям
   - Критерии достижения явные и проверяемые
   - Заинтересованные стороны идентифицированы
   - Требования конкретны, без общих фраз
   - Риски осознаны

2. **Сокращение времени подготовки БТ** по сравнению с текущим процессом

3. **Документированная оценка трудозатрат:** оценка становится воспроизводимой, а не «знанием в голове» при уходе человека

4. **Повышение культуры работы с БТ:** каждая новая инициатива в Jira содержит БТ, подготовленный сервисом

### Key Performance Indicators

| Метрика | Цель | Измерение |
|---------|------|-----------|
| Полнота разделов БТ | 7/7 блоков заполнены | Автоматическая проверка контролёром |
| Глубина проработки | Экспертная оценка «хорошо» и выше | Качественная оценка (не автоматизирована) |
| Время подготовки БТ | Меньше, чем ручной процесс | Сравнение с текущими трудозатратами |
| Доля инициатив с БТ | 100% новых инициатив | Jira-задачи с прикреплённым БТ |
| Удовлетворённость пользователя | Легко и понятно проходить опрос | Обратная связь от PO и БА |
| Улучшение от сессии к сессии | Качество растёт | Анализ паттернов вопросов (самообучение) |

## MVP Scope

### Core Features

1. **@Sbernasledtsvo_bot** (через OpenClaw gateway, команда /создатьбт) как основной канал взаимодействия
2. **Динамический опрос** по 7 блокам утверждённого шаблона БТ:
   - Агент последовательно проходит по разделам
   - Вопросы адаптируются под ответы пользователя
   - При поверхностном ответе — уточняющие вопросы

3. **Мультиагентный пайплайн:**
   - Агент-опросчик — сбор информации через диалог
   - Агент-составитель — формирование текста по шаблону
   - Агент-контролёр — проверка полноты и глубины разделов
   - Агент-генератор — создание DOCX и веб-версии

4. **Генерация DOCX** (python-docx на VPS)
5. **Публикация на GitHub Pages** (DOCX + веб-страница)
6. **Хранение истории сессий** (черновики, завершённые БТ, паттерны)
7. **Самообучаемость:** накопление опыта для улучшения качества вопросов

### Out of Scope for MVP

- Интеграция с Jira (пользователь прикрепляет DOCX самостоятельно)
- Мультиязычность (только русский язык)
- Агрегация метрик качества (оценка экспертная, не автоматизированная)
- Party Mode и Advanced Elicitation (продвинутые режимы опроса)
- Веб-интерфейс для просмотра сессий (только GitHub Pages для готовых документов)

### MVP Success Criteria

- Сервис успешно проводит опрос по всем 7 блокам
- Контролёр возвращает на доработку поверхностные разделы
- DOCX-файл генерируется корректно
- Веб-версия публикуется на GitHub Pages
- Пользователь получает ссылку в Telegram

### Future Vision

- **2-3 года:** Интеграция с Jira для автоматического прикрепления БТ к задачам
- Автоматическая агрегация метрик качества
- Расширение на английский язык
- Веб-интерфейс для управления сессиями
- API для интеграции с другими системами
- Аналитическая панель с метриками по зрелости БТ
