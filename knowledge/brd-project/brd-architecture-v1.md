## 1. Общая схема пайплайна

```
[Пользователь: «Создать БТ»]
        │
        ▼
┌─────────────────────────────────────────────┐
│  Controller-агент (оркестратор)             │
│  • Точка входа: триггер «Создать БТ»        │
│  • Управляет жизненным циклом BRD           │
│  • Держит состояние сессии                  │
└──────────┬──────────────────────────────────┘
           │
      ┌────┴──── sessions_send ────┐
      │                              │
      ▼                              ▼
┌──────────────────┐       ┌──────────────────────┐
│  Questioner-агент │       │  Compiler-агент       │
│  (биндинг к чату)│       │  (sessions_send)      │
└────────┬─────────┘       └──────────┬───────────┘
         │                             │
         │  Лог интервью (.md)         │  Драфт BRD (.md)
         │                             │
         └──────────┬──────────────────┘
                    │
                    ▼
         ┌──────────────────────┐
         │  Verifier-агент      │
         │  (sessions_send)     │
         └──────────┬───────────┘
                    │
              ┌─────┴─────┐
              │           │
              ▼           ▼
          [Итерация]   [✅ Аппрув]
              │           │
              └→ Compiler │
                          ▼
              ┌──────────────────────┐
              │  Controller:          │
              │  .md → .docx          │
              │  → отправка в чат     │
              └──────────────────────┘
```

## 2. Протокол передачи между агентами

### 2.1 Формат .md-лога (единый межсессионный протокол)

Каждый агент принимает на входе и выдаёт на выходе `md_log` — структурированный Markdown-документ.

```markdown
# BRD Session: <UUID-сессии>
## Meta
- user: <telegram_id>
- created: 2026-07-14T09:00:00+03:00
- status: interview_complete | compiled | verified | approved | rejected
- iteration: 0 | 1 | 2
- controller_session: <sessionKey Controller>
- questioner_session: <sessionKey Questioner>

## QuestionLog
### Q1: <вопрос>
**A:** <ответ пользователя>

### Q2: <вопрос>
**A:** <ответ пользователя>

...

### Summary
- Симптом: <формулировка>
- Барьер: <формулировка>
- Потери: <цифры/оценка>
- Контекст: <дополнительно>

## CompiledBRD
### Description
[Симптом и измеримые Потери] происходит из-за [Барьер], что вызвано корневой причиной: [Корень].

### Goal
<SMART-цель>

### Metrics
<ключевые метрики успеха>

### Impacts
<impact mapping>

### References
<ссылки из Web Search>

## VerificationLog
### Check: RCA_Integrity
- Status: ✅ / ❌ / 🔄
- Comment: <что не так>

### Check: SMART
- Status: ✅ / ❌ / 🔄
- Comment: <что не так>

### Check: Linguistic
- Status: ✅ / ❌ / 🔄
- Comment: <что не так>

### Verdict: APPROVED / REVISION_NEEDED
- RevisionCount: <текущая>
- MaxRevisions: 2
```

### 2.2 Схема передачи

| Шаг | От | Кому | Что передаётся | Что получает обратно |
|-----|----|------|----------------|---------------------|
| 1 | Controller | Questioner | md_log (пустой, только Meta) | md_log (заполнен QuestionLog) |
| 2 | Controller | Compiler | md_log (с QuestionLog) | md_log (добавлен CompiledBRD) |
| 3 | Controller | Verifier | md_log (с QuestionLog + CompiledBRD) | md_log (добавлен VerificationLog) |
| 4 (если ❌) | Controller | Compiler | md_log (с VerificationLog, iteration+1) | md_log (обновлён CompiledBRD) |
| 5 (повтор 3) | Controller | Verifier | md_log (обновлён) | md_log (финальный) |

## 3. SKILL.md для каждого агента

### 3.1 Controller-agent — SKILL.md

```markdown
# Controller — Оркестратор BRD

## Триггер
- Сообщение пользователя содержит «Создать БТ»

## Жизненный цикл
1. Создать UUID сессии
2. Заполнить Meta в md_log
3. sessions_send(agentId="questioner-ba", message=md_log) → ждать ответ
4. sessions_send(agentId="compiler-ba", message=md_log) → ждать ответ
5. sessions_send(agentId="verifier-ba", message=md_log) → ждать ответ
6. Если Verifier вернул REVISION_NEEDED → вернуться к шагу 4 (макс 2 итерации)
7. Конвертировать md_log в .docx (через python-docx)
8. Отправить .docx в чат + краткую сводку

## Экспорт .docx
- Шаблон: python-docx, секции: Проблема, Цель, Метрики, Импакты, Источники
- Файл: bt_<дата>_<id>.docx
- Отправка: прикрепить к сообщению Telegram

## Таймауты
- Questioner: 600 сек (интервью с пользователем может быть долгим)
- Compiler: 120 сек
- Verifier: 90 сек
- Итерация: макс 2 ревизии
```

### 3.2 Questioner-agent — SKILL.md

```markdown
# Questioner — Старший бизнес-аналитик

## Биндинг
- В тот же чат, что и пользователь
- Не реагирует на сообщения без активной сессии

## Активация
- Получает md_log от Controller через sessions_send
- Начинает диалог с пользователем в чате

## Правило №1: Не верь пользователю
- 99% пользователей не знают свою проблему
- Игнорировать запросы «сделайте мне фичу» — вытаскивать контекст процесса
- Подозревать: то, что названо причиной — симптом

## Методика опроса
1. **Симптом** → «Что конкретно происходит? Что ты видишь/слышишь?»
2. **Барьер** → «Что мешает? Где застреваешь?»
3. **Потери (в цифрах)** → «Сколько времени/денег/людей теряешь? В день/неделю/месяц?»
4. **5 Whys (мягкая версия)** → «Что происходит сразу после того, как...»

## Правила коммуникации
- Анонс длительности на старте: «Я задам 5-20 вопросов, это займёт ~10-15 минут»
- Микро-резюме каждые 3 вопроса: «Пока я понял так: ... Я правильно понимаю?»
- Право на «стоп»: «Если нужно прерваться — скажи «стоп», я сохраню результат»
- Благодарить за каждый ответ

## Выход
- Заполняет md_log → добавляет QuestionLog + Summary
- Отправляет md_log обратно Controller-у
- Завершает диалог: «Спасибо! Передаю аналитику. Результат приду сюда же.»

## Таймаут
- При бездействии пользователя > 15 минут: сохранить что есть и завершить
```

### 3.3 Compiler-agent — SKILL.md

```markdown
# Compiler — Системный архитектор BRD

## Активация
- Получает md_log с заполненным QuestionLog от Controller через sessions_send

## Задачи
1. **RCA — корневая причина**
   - Из Summary (Симптом → Барьер → Потери) вывести корневую причину
   - Формула: «[Симптом и измеримые Потери] происходит из-за [Барьер], что вызвано корневой причиной: [Корень].»

2. **Web Search (через web_search)**
   - Искать: лучшие практики, аналогичные кейсы, технологические решения
   - 2-3 релевантных источника добавить в References

3. **SMART-цель**
   - Specific, Measurable, Achievable, Relevant, Time-bound
   - Жёсткая привязка к RCA

4. **Метрики**
   - Количественные метрики успеха (верифицируемые)
   - Baseline → Target

5. **Impact Mapping**
   - Цель → Действия → Результаты → Метрики

## Few-Shot (2 примера)
- Пример 1: Чат-бот для поддержки
  - Симптом: «Операторы отвечают 2 часа»
  - Корень: «Нет self-service, все вопросы идут в 1st line»
  - Цель: «Время ответа < 5 мин для 80% запросов за 3 мес»

- Пример 2: Интерфейс бухгалтера
  - Симптом: «Закрытие месяца занимает 5 дней»
  - Корень: «Ручной ввод данных из 3 систем»
  - Цель: «Закрытие месяца за 1 день через 6 мес»

## Выход
- Добавляет CompiledBRD в md_log
- Отправляет md_log обратно Controller-у
```

### 3.4 Verifier-agent — SKILL.md

```markdown
# Verifier — Логический аудитор / Red Teaming

## Активация
- Получает md_log с CompiledBRD от Controller через sessions_send

## 3 проверки (строго, без пропусков)

### 1. RCA Integrity
- Проблема → Цель → Метрика как единое уравнение
- Пример прохода: «Операторы отвечают 2 часа» → «Время ответа < 5 мин» → «p95 latency / SLA %»
- Если связь рвётся — ❌

### 2. Симптом ≠ Корень (охота на человеческий фактор)
- Если корневая причина = «не хватает рук»,  «плохая мотивация», «человеческий фактор» — ❌
- Системная причина = процесс, инструмент, данные, организация, а не люди

### 3. Лингвистический комплаенс
- В Description ОБЯЗАТЕЛЬНО наличие связок:
  - «происходит из-за»
  - «что вызвано корневой причиной»
- и всех трёх элементов: Симптом, Барьер, Корень
- Если нет — ❌

## Вердикт
- Все ✅ → APPROVED
- Любой ❌ → REVISION_NEEDED + Comment (что именно не так)

## Ревизии
- Макс 2 итерации
- На 3-й раз → APPROVED_FORCED с пометкой «forced after 2 revisions»
```

## 4. Схема вызовов (детальная)

### Первичный прогон
```
1. Controller: sessions_send(agentId="questioner-ba", message=md_log_meta)
   → Controller ждёт (таймаут 600 сек)
2. Questioner: начинает диалог, собирает ответы
   → Questioner: sessions_send(Controller, md_log_question)
3. Controller: sessions_send(agentId="compiler-ba", message=md_log_question)
   → Controller ждёт (таймаут 120 сек)
4. Compiler: RCA + Web Search + Impact Mapping
   → Compiler: sessions_send(Controller, md_log_compiled)
5. Controller: sessions_send(agentId="verifier-ba", message=md_log_compiled)
   → Controller ждёт (таймаут 90 сек)
6. Verifier: 3 проверки → вердикт
   → Verifier: sessions_send(Controller, md_log_verified)
```

### Итерация (если Verifier ❌)
```
7. Controller: iteration++ (макс 2)
8. Controller: sessions_send(agentId="compiler-ba", message=md_log_revision)
9. Compiler: исправляет по комментариям Verifier-а
10. → Шаг 5 (Verifier)
```

### Финал
```
11. Controller: md_log[status=approved] → .docx
12. Controller: отправка файла в чат + текст:
    «✅ БТ готов: bt_2026-07-14_<id>.docx
    Проблема: <корень>
    Цель: <SMART>
    Метрики: <KPI>»
```

## 5. Точки отказа и обработка ошибок

| Точка отказа | Симптом | Действие Controller |
|-------------|---------|-------------------|
| Questioner не отвечает > 600 сек | Timeout | Отправить пользователю: «Интервью прервалось. Начни заново — напиши «Создать БТ»» |
| Questioner завис в диалоге | Нет финального md_log | Fallback: вопросы сохранены частично, Compiler работает с тем что есть |
| Questioner промолчал / не начал диалог | Пользователь не получил вопросов | Controller: перезапустить Questioner 1 раз |
| Compiler не отвечает > 120 сек | Timeout | Fallback: web_search исключён, RCA на основе только Summary |
| Compiler завис | Нет CompiledBRD | Controller: повторить 1 раз через 30 сек |
| Verifier не отвечает > 90 сек | Timeout | Пропустить верификацию, выдать как есть с пометкой «не верифицирован» |
| Compiler не может исправить после 2 итераций | Verifier снова ❌ | APPROVED_FORCED + пометка |
| python-docx ошибка при конвертации | Нет файла | Controller: выдать .md лог как текст в чат |
| Ошибка прикрепления файла в Telegram | Файл не отправился | Controller: опубликовать на GitHub Pages + ссылка |
| Нет активной сессии Questioner-а | sessions_send не находит агента | Controller: уведомить пользователя «Система временно недоступна, попробуйте позже» |

### Глобальный таймаут сессии BRD
- 30 минут с момента «Создать БТ»
- Если не уложились → abort, пользователю: «Сессия прервана по таймауту. Начните заново.»

| Параметр | Questioner | Compiler | Verifier | Итерация |
|----------|------------|----------|----------|----------|
| Max response time | 600 сек | 120 сек | 90 сек | +120 сек |
| Max total: | 600 сек | 240 сек (2 итерации) | 180 сек (2 итерации) | |
| **Итого макс:** | | **1020 сек = 17 мин** | | |
| Retry policy | 1 раз | 1 раз | нет | |
| Fallback behaviour | частичные данные | без web_search | пропуск верификации | |
