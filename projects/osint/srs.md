## 1. Архитектура системы

### 1.1. Общая схема (фазная)

```
[Фаза A — Сбор в 05:30/13:30/19:30]
  web_fetch(ISW, Guardian, Al Jazeera, pravo.gov.ru) + SearXNG
  → osint-raw-data.json (сырые данные)
          ↓
[Фаза B — Анализ + Генерация в 05:50/13:50/19:50]
  LLM-анализ → верификация → HTML-генерация
  → osint.html (локально) + osint-seen.md (обновлён)
          ↓
[Фаза C — Публикация в 05:55/13:55/19:55]
  git add → commit → push → GitHub Pages
```

### 1.2. Компоненты

| Компонент | Роль | Технология |
|-----------|------|------------|
| **Skill: osint-expert** | Инструкция для cron-сессии: форматы, источники, система оценок, CSS-классы | Skill Workshop |
| **Задача A: Сбор данных** | Парсинг источников через web_fetch + SearXNG, запись в JSON | isolated cron-сессия, 180s таймаут |
| **Задача B: Анализ + HTML** | Синтез фактов, верификация, формирование osint.html | isolated cron-сессия, deepseek-chat, 300s таймаут |
| **Задача C: Публикация** | git add → commit → push | isolated cron-сессия, 60s таймаут |
| **Memory/Session** | Дедупликация, логирование, промежуточные файлы | memory/osint/ (osint-seen.md, osint-log.md, osint-raw-data.json) |

### 1.3. Взаимодействие

Каждая фаза — отдельная isolated cron-задача с независимым таймаутом:
1. **Фаза A** (05:30/13:30/19:30): опрос источников → `osint-raw-data.json`
2. **Фаза B** (05:50/13:50/19:50): чтение raw-data → анализ → верификация → генерация HTML → `osint.html` + обновление `osint-seen.md`
3. **Фаза C** (05:55/13:55/19:55): чтение osint.html → git add/commit/push

### 1.4. Изоляция

Проект OSINT работает в собственных isolated cron-сессиях, не влияя на:
- Основную сессию Telegram
- Другие cron-задачи (Катя, Лена, Данил, Катрин)
- AI DJ / другие сервисы

## 2. Сценарии использования (Use Cases)

### UC-1: Просмотр дайджеста
- **Описание:** Пользователь открывает страницу osint.html и видит текущие оценки вероятности по трём вопросам с фактологическим обоснованием
- **Акторы:** Любой пользователь (без авторизации)
- **Предусловия:** Страница опубликована на GitHub Pages
- **Результат:** Пользователь ознакомился с текущими оценками и подтверждающими фактами
- **Исключения:** Страница недоступна — ошибка 404/500

### UC-2: Сбор данных (системный)
- **Описание:** Фаза A cron-задачи опрашивает источники и сохраняет сырые данные
- **Акторы:** Система (isolated cron-сессия)
- **Предусловия:** Доступны основные источники
- **Результат:** osint-raw-data.json обновлён
- **Исключения:** Недоступность источника (фиксируется в логе, дайджест собирается по доступным)

### UC-3: Анализ и генерация (системный)
- **Описание:** Фаза B читает сырые данные, анализирует, верифицирует и формирует HTML
- **Акторы:** Система (isolated cron-сессия)
- **Предусловия:** osint-raw-data.json существует и не пуст
- **Результат:** osint.html обновлён локально, osint-seen.md пополнен
- **Исключения:** Сбой LLM — дайджест не генерируется

### UC-4: Публикация (системный)
- **Описание:** Фаза C публикует osint.html на GitHub Pages
- **Акторы:** Система (isolated cron-сессия)
- **Предусловия:** osint.html существует, git настроен
- **Результат:** osint.html обновлён на GitHub Pages
- **Исключения:** Ошибка git push — уведомление Кириллу

Сценарий администрирования не предусмотрен — проект полностью автоматизирован.

## 4. Требования к данным

### 4.1. История дайджестов
История дайджестов не хранится. Каждый новый дайджест полностью заменяет предыдущий на странице osint.html.

**Обоснование:** задачи аналитики по динамике изменений на старте нет. В будущем при необходимости аналитики трендов можно добавить хранение истории в формате JSON-лога (файл `memory/osint/osint-history.json` — append-only).

### 4.2. Дедупликация
Обязательна. Алгоритм:
- Каждый факт/новость проверяется по комбинации: источник + дата + ключевое утверждение
- Файл дедупликации: `memory/osint/osint-seen.md` (как у Лены `lena-news-seen.md`)
- Если факт уже встречался в предыдущем дайджесте и не изменился — не включается повторно
- Если факт обновился (новая дата, новая деталь) — включается с пометкой «обновлено»

### 4.3. Форматы данных
| Данные | Формат | Назначение |
|--------|--------|------------|
| Выходной дайджест | HTML (osint.html) | Публикация на GitHub Pages |
| Сырые данные (промежуточные) | JSON (osint-raw-data.json) | Передача между фазами A→B |
| Дедупликация | Markdown (osint-seen.md) | Хранение истории фактов |
| Лог операций | Markdown (osint-log.md) | Мониторинг работы |

### 4.4. Жизненный цикл
1. **Фаза A — Сбор** → osint-raw-data.json (сырые данные из источников)
2. **Фаза B — Анализ** → osint.html + osint-seen.md (дедупликация)
3. **Фаза C — Публикация** → git push на GitHub
4. **Очистка** — osint-raw-data.json перезаписывается при следующей фазе A

## 6. Функциональные требования

### FR-1: Сбор данных из источников
Система должна собирать данные из всех категорий источников (раздел 5 БТ) с помощью доступных инструментов (web_fetch, SearXNG, прямые API).

### FR-2: Анализ и синтез фактов
LLM должна анализировать собранные данные, выделять ключевые факты и синтезировать их в связную картину по трём вопросам.

### FR-3: Верификация
Агент-верификатор должен выполнять перекрёстную проверку фактов из независимых источников перед включением в дайджест.

### FR-4: Дедупликация
Система должна проверять каждый факт на уникальность (источник + дата + ключевое утверждение) перед добавлением в дайджест.

### FR-5: Оценка вероятности
Для каждого из трёх вопросов система должна выставить оценку: Высокая (🔴), Средняя (🟡), Низкая (🟢).

### FR-6: Генерация HTML
Система должна сформировать HTML-страницу osint.html с карточками, цветовыми метками и фактологическим обоснованием по утверждённому шаблону.

### FR-7: Публикация на GitHub Pages
Сгенерированная страница должна быть опубликована через git push на https://nasledstvo2026.github.io/nasledstvo/osint.html.

### FR-8: Обработка ошибок
Система должна:
- Фиксировать недоступность источников
- Обрабатывать противоречивые данные
- Не публиковать дайджест при критических сбоях
- Уведомлять Кирилла при повторяющихся ошибках

## 7. Нефункциональные требования

### 7.1. Производительность
- Время сборки дайджеста — сумма 3 фаз (A+B+C), каждая с независимым таймаутом
- Фаза A (сбор): ≤ 180 сек
- Фаза B (анализ + HTML): ≤ 300 сек
- Фаза C (публикация): ≤ 60 сек
- Общий цикл: ≤ 540 сек (9 минут), при 20-минутных зазорах между фазами

### 7.2. Доступность
- Доступность 24/7 не требуется
- Допустимы периоды недоступности при сбоях (недоступность источников, ошибки LLM, проблемы git push)
- Цель: публикация дайджеста по расписанию ≥ 80% времени (не жёсткое требование)

### 7.3. Безопасность
- Все данные — из публичных источников (OSINT)
- Никакой приватной информации не обрабатывается и не хранится
- Авторизация на странице не требуется

### 7.4. Надёжность
- failureAlert: после 2 ошибок — уведомление Кириллу (cool down 1 час), действует на каждую фазу независимо
- При недоступности 2+ ключевых источников — дайджест не публикуется
- При сбое LLM — автоматический повтор с fallback-моделью
- Сбой публикации (фаза C) не влияет на данные — osint.html остаётся локально, git push можно повторить вручную

## 9. Требования к мониторингу и логированию

### 9.1. Health-check
- **Периодичность:** во время каждого запуска сборки дайджеста
- **Проверка:** доступность ключевых источников, работоспособность LLM API, git push
- При недоступности источника — пропуск, дайджест собирается по доступным данным
- При критическом сбое (LLM API, git) — дайджест не публикуется, уведомление Кириллу

### 9.2. Логирование
- Каждая операция логируется: старт сбора, результаты опроса источников, факты до/после верификации, генерация HTML, публикация
- Формат логов: текстовый файл `memory/osint/osint-log.md`
- Каждая запись: метка времени (`[YYYY-MM-DD HH:MM]`) + операция + результат

### 9.3. Управление логами
- **Глубина хранения:** 1 день
- **Обнуление:** ежедневно перед стартом сборки дайджеста (старый лог удаляется, создаётся новый)

### 9.4. Уведомления об ошибках
- **Кому:** Кириллу (@Kirill_syst)
- **Когда:** после 2 последовательных ошибок (failureAlert, cool down 1 час)

## 10. Skill osint-expert

### 10.1. Назначение
Skill `osint-expert` — шаблонная инструкция для isolated cron-сессии. Определяет:
- Категории источников и приоритет их опроса
- Формат дайджеста (разделы, карточки, CSS-классы)
- Систему оценки вероятности (Высокая/Средняя/Низкая)
- Принципы атрибуции фактов
- Алгоритм дедупликации

### 10.2. Как используется
1. При старте cron-задачи isolated-сессия читает скилл
2. На основе инструкций выполняет сбор, анализ, верификацию
3. Формирует HTML строго по шаблону из скилла

### 10.3. Преимущества
- Единый стандарт для всех запусков (независимо от модели)
- Меньше токенов в теле cron-промпта
- Централизованное обновление: меняешь скилл → меняется поведение всех запусков

## 11. Саморазвитие агентов

### 11.1. На старте (MVP)
Саморазвитие **не закладывается**. Система работает по фиксированному алгоритму:
- Сбор → Анализ → Верификация → Публикация
- Каждый цикл независим от предыдущего
- Единственная память между циклами — файл дедупликации osint-seen.md

### 11.2. В будущем (Roadmap)
Саморазвитие возможно через:
- **Итеративное улучшение промптов:** предыдущий дайджест + выявленные ошибки → коррекция промпта следующего цикла
- **Накопление знаний:** факты, подтверждённые верификатором, формируют базу «подтверждённых паттернов»
- **Обратная связь:** если пользователь указал на ошибку — кейс запоминается и учитывается в будущих оценках

Решение о внедрении саморазвития — после оценки стабильности MVP.

## 13. Интеллектуальный движок оценки вероятности

### 13.1. Назначение
Ключевой элемент системы, отвечающий за минимизацию ошибок и формирование объективной оценки вероятности по трём вопросам.

### 13.2. Архитектура движка
```
[Сбор данных] → [Фильтрация шума] → [Синтез фактов] → [Верификация] → [Оценка вероятности] → [Формирование вывода]
```

### 13.3. Этапы работы

**Фаза A: Сбор и фильтрация** (cron-задача, 180s)
- Сборщик опрашивает все доступные источники через web_fetch + SearXNG
- Отсеиваются явно недостоверные/пропагандистские материалы
- Результат: osint-raw-data.json — массив верифицируемых фактов с атрибуцией
- Логирование в osint-log.md

**Фаза B: Синтез → Верификация → Генерация** (cron-задача, 300s)
- **Этап 1: Синтез** — LLM (deepseek-chat) анализирует массив фактов из osint-raw-data.json, выделяет ключевые паттерны и тренды по каждому из 3 вопросов, формирует предварительную оценку с обоснованием
- **Этап 2: Верификация** — LLM перекрёстно проверяет: для каждого факта ищет подтверждение/опровержение в независимых источниках. Если факт подтверждается < 2 независимыми источниками — помечается как «недостаточно подтверждён». Если факт опровергается — исключается из обоснования
- **Этап 3: Формирование оценки** — Итоговая оценка выставляется по совокупности подтверждённых фактов:
  - Большинство фактов указывают на событие → Высокая (🔴)
  - Факты противоречивы или недостаточны → Средняя (🟡)
  - Большинство фактов против → Низкая (🟢)
- **Этап 4: Генерация HTML** — формирование osint.html по шаблону
- **Дедупликация** — сверка с osint-seen.md, добавление новых фактов
- Логирование в osint-log.md

**Фаза C: Публикация** (cron-задача, 60s)
- git add osint.html → git commit → git push
- Логирование в osint-log.md

### 13.4. Принципы работы
- **Никакой пропаганды:** оценка базируется только на фактах, не на эмоциях или политических установках
- **Прозрачность:** пользователь видит не только цвет, но и факты, на которых он основан
- **Атрибуция:** каждый факт → конкретный источник + дата
- **Минимизация ошибок:** двойная проверка (LLM → верификатор) снижает риск единичной ошибки

### 13.5. Обработка противоречий
- Если источники дают взаимоисключающую информацию — верификатор собирает оба мнения с атрибуцией
- Итоговая оценка учитывает: авторитетность источника, количество источников, свежесть данных
- При невозможности сделать однозначный вывод — оценка «Средняя» (🟡) с пометкой «данные противоречивы»
