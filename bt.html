💡 Что такое «Создать БТ»

«Создать БТ» — мультиагентный сервис для формирования бизнес-требований (БТ) высокого уровня зрелости. Сервис решает проблему низкой культуры работы с БТ: владельцы продуктов и бизнес-аналитики тратят много времени, используют разные шаблоны, формулируют поверхностные требования без анализа корневых причин. «Создать БТ» через динамический опрос и мультиагентный контроль помогает создавать глубокие, структурированные и измеримые бизнес-требования.

🔬 Ключевые возможности

🤖
Динамический опрос
Агент задаёт вопросы в Telegram, адаптируясь под ответы. При поверхностном ответе — копает глубже, до корневой причины.
🔄
Мультиагентный пайплайн
4 агента: опросчик → составитель → контролёр → генератор. Контролёр возвращает на доработку, если раздел заполнен поверхностно.
📄
DOCX + веб-версия
Двойной формат выдачи: DOCX для Jira + веб-страница на GitHub Pages для просмотра и шаринга.
🧠
Самообучаемость
С каждым новым БТ качество опроса повышается: сервис запоминает паттерны, которые вели к хорошим требованиям.

📋 Шаблон БТ (7 блоков, неменяемый)

Агент-опросчик проходит блоки последовательно, с динамической глубиной (L1 → L2 → L3):

Блок 1
📌 Контекст и проблема
1.1 Контекст инициативы · 1.2 Проблемная область · 1.3 Последствия бездействия
L1 «Расскажите в двух словах…» L2 «Приведите пример…» L3 «Какая коренная причина?»
Блок 2
🎯 Цель и измерение
2.1 Бизнес-цель · 2.2 Критерии успеха
L1 «Какую бизнес-цель преследуете?» L2 «Сформулируйте измеримо» L3 «Какие целевые KPI?»
Блок 3
👥 Стейкхолдеры и влияние
3.1 Заинтересованные стороны · 3.2 Их интересы и влияние
L1 «Кто затронут?» L2 «По группам: пользователи, внедренцы…» L3 «Конфликт интересов?»
Блок 4
🛠 Решение
4.1 Ключевые функции · 4.2 Пользовательские сценарии · 4.3 Граничные случаи
L1 «Что решение должно делать?» L2 «Must/Should/Nice to have» L3 «Для каждой must have — что кодить?»
Блок 5
⚡ Требования к качеству и ограничения
5.1 Качественные характеристики · 5.2 Ограничения
L1 «Производительность, безопасность?» L2 «Сколько пользователей? Аутентификация?» L3 «99.9% uptime? RTO < 1 час?»
Блок 6
✅ Приёмка
6.1 Критерии приёмки
L1 «Как поймём, что функция реализована?» L2 «Given-When-Then» L3 «Кто принимает? Автотесты?»
Блок 7
⚠️ Риски и митигации
7.1 Идентифицированные риски · 7.2 Митигации
L1 «Систематизируем риски» L2 «Технические, организационные, adoption» L3 «Для критических: план Б?»

🛠 Техническая основа

  • Платформа: OpenClaw (изолированные сессии агентов)
  • Модель: DeepSeek Chat (deepseek/deepseek-chat)
  • Канал: Telegram
  • Архитектура: мультиагентный пайплайн (4 агента, agent-to-agent)
  • Генерация DOCX: python-docx (на VPS)
  • Публикация: GitHub Pages (веб-версия + DOCX-файл)
  • Хранение: история сессий на VPS (черновики, завершённые БТ, паттерны для самообучения)
  • Язык: только русский

📐 Критерии зрелости БТ

Сервис достиг цели, если подготовленный БТ соответствует критериям:

  • ✅ Проблемная область — проанализирована до корневой причины
  • ✅ Бизнес-цель — измерима, привязана к KPI
  • ✅ Критерии успеха — явные, достижимые, проверяемые
  • ✅ Стейкхолдеры — все идентифицированы, интересы учтены
  • ✅ Функции — конкретны, без общих фраз
  • ✅ NFR и ограничения — зафиксированы
  • ✅ Приёмка — проверяемые критерии
  • ✅ Риски — осознаны, а не отписка

KPI: Полнота 7/7 блоков · Глубина «хорошо» и выше · Время подготовки меньше ручного · Доля инициатив с БТ = 100%

📦 MVP Scope

Core Features:

  • Telegram-бот — основной канал
  • Динамический опрос по 7 блокам шаблона с L1→L2→L3
  • Мультиагентный пайплайн (опросчик → составитель → контролёр → генератор)
  • Генерация DOCX (python-docx на VPS)
  • Публикация на GitHub Pages (DOCX + веб-страница)
  • Хранение истории сессий
  • Самообучаемость

Out of Scope: Jira-интеграция, мультиязычность, веб-интерфейс сессий, агрегация метрик

🚫 Что сервис НЕ делает

  • Не прикрепляет БТ в Jira (пользователь делает это сам)
  • Не пишет код и не проектирует архитектуру
  • Не работает на английском (только русский)
  • Не даёт веб-интерфейс для управления сессиями
  • Не считает автоматическую оценку качества БТ (оценка экспертная)
  • Не заменяет Product Owner — сервис ассистирует, а не решает

📱 Как пользоваться

  1. Напиши в Telegram — сервис начнёт опрос
  2. Отвечай на вопросы свободно — агент адаптируется под твой контекст
  3. При поверхностном ответе агент задаст уточняющие вопросы
  4. После прохождения всех 7 блоков — дождись проверки контролёра
  5. Получи готовый DOCX + ссылку на веб-версию в Telegram
  6. Прикрепи DOCX к Jira-задаче

⏱ Типичное время опроса: 10–25 минут, в зависимости от сложности инициативы.
