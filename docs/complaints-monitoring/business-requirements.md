# Бизнес-требования: Мониторинг жалоб по наследству в банках РФ

**Проект:** Цифровой продукт «Мониторинг жалоб»
**Версия:** 2.0 (реверс-инжиниринг)
**Дата:** 02.07.2026
**Автор:** Лунт (Business Analyst)
**Статус:** Утверждено (по факту реализации)

---

## 1. Executive Summary

Цифровой продукт обеспечивает ежедневный автоматический мониторинг, сбор, фильтрацию, категоризацию и аналитику жалоб граждан по наследованию денежных средств в банках РФ.

Система работает в полностью автоматическом режиме: поиск через SearXNG, анализ через LLM (DeepSeek), публикация отчёта на GitHub Pages, отправка сводки аналитику в Telegram. Резервный запуск на случай сбоя основного.

**Масштаб:** 77 жалоб в базе (январь 2025 — июнь 2026), 19 банков, 4 площадки-источника.

---

## 2. Бизнес-контекст и проблема

### 2.1. Проблема
Граждане-наследники сталкиваются с отказами, задержками и бюрократическими препятствиями со стороны банков РФ при получении наследственных денежных средств (вклады, счета, акции). Проблема носит массовый характер.

Аналитик Катя тратила часы на ручной поиск жалоб по разрозненным площадкам. Без автоматизации:
- Часть жалоб пропускалась
- Отсутствовала системная статистика по банкам-нарушителям
- Невозможно было отследить динамику

### 2.2. Решение
Автоматизированный пайплайн: SearXNG (поиск) → LLM (анализ, фильтрация, категоризация) → Telegram (сводка) + GitHub Pages (статистика).

### 2.3. Ценность для бизнеса
- Экономия времени аналитика: с 30+ минут ручного поиска до 0 (сводка приходит сама)
- Системная статистика: какие банки чаще нарушают, какие типы проблем преобладают
- База для адвокации: накопленные данные можно использовать в обращениях

### 5.4. Спецификация шаблона статистической страницы (stats-inheritance-template.html)

Статистическая страница строится по жёсткому HTML-шаблону `stats-inheritance-template.html`. LLM-агент обновляет только текстовые данные; структура, стили, классы, id и расположение элементов НЕ МЕНЯЮТСЯ.

#### 5.4.1. Критические ограничения шаблона

| TL# | Ограничение | Описание | Причина |
|-----|-------------|----------|---------|
| TL-01 | **Светлая тема** | Цветовая схема `--bg: #f5f7fa`, `--glass-bg: rgba(255,255,255,0.85)` | Никогда не менять на тёмную |
| TL-02 | **Классы, id, теги — заморожены** | LLM не добавляет и не удаляет классы/атрибуты | Сохранение вёрстки |
| TL-03 | **Только текстовые замены** | Меняется только текстовое содержимое тегов. Все атрибуты (href, style, class, onclick и т.д.) сохраняются | Целостность шаблона |
| TL-04 | **Версионирование** | Найти тег `v{N}` в footer и увеличить N на 1 | Отслеживание изменений |

#### 5.4.2. Секции шаблона (порядок строгий)

| № | Секция | Класс / Идентификатор | Содержимое | Источник |
|---|--------|----------------------|------------|----------|
| 1 | **Шапка (Hero)** | `.hero` → `<h1>` | Заголовок страницы | Статичен (не меняется) |
| 2 | **Вчерашний день** | `.section` → `.item` → `.tag[📅]` | Дата (текстовая), счётчики «Сбер» и «Другие банки» (цвет: 0 = green, >0 = red), ссылки на жалобы с url | katya-data.json, фильтр по вчерашней дате |
| 3 | **Текущий месяц** | `.section` → `.item` → `<h2>📊 {Месяц} {Год}</h2>` | Всего жалоб, Сбер/Другие (inline-блоки), таблица динамики (предыдущий месяц → текущий, Δ), список банков за месяц | katya-data.json, фильтр по текущему году-месяцу |
| 4 | **Банки за год** | `.section` → `.item` → `<h2>🏦 Банки · {Год}</h2>` | Горизонтальные бары: `.bar-fill.{bank}`, ширина относительно Сбера (max 230px), проценты от общего количества | katya-data.json, фильтр по текущему году |

| 6 | **Хронология** | `.section` → `.item` → `<h2>⚠️ Хронология</h2>` | Таблица: «Дата», «Банк», «Описание», «Источник». Сортировка по убыванию даты. Ссылки: url непустой = `<a href="..." target="_blank">`, url пустой = просто текст | katya-data.json, все записи |
| 7 | **Примечание** | `.item` (предпоследний элемент body) | «Данные собраны {дата}» | Текущая дата запуска |
| 8 | **Footer** | `.footer` | «Обновлено {дата} v{N}» | Текущая дата + инкремент версии |

#### 5.4.3. Цвета баров (зафиксированы в CSS шаблона)

| CSS-класс | Банк | HEX-цвет |
|-----------|------|----------|
| `.sber` | Сбер | `#1a9e5c` |
| `.vtb` | ВТБ | `#009fdf` |
| `.psb` | ПСБ | `#f15a24` |
| `.tbank` | Т-Банк | `#ffdd3c` |
| `.gpb` | Газпромбанк | `#003a70` |
| `.sovcombank` | Совкомбанк | `#d42027` |

Для остальных банков (Альфа-Банк, Райффайзен, Ozon Банк, МТС Банк, Неизвестно, и т.д.) — inline-style `background: <HEX>`.

**Правила:**
- Ширина бара Сбера = 230px (100%). Остальные барьи — пропорционально количеству жалоб
- Проценты = `(count / total) × 100`, один знак после запятой
- Формат подписи: `Название (12.3%)`

#### 5.4.4. Правила таблиц

**Секция «Статистика»:**
- 5 колонок: Месяц | Сбер | Банки | Итог | Банк
- Итоговая строка года:
  - 2026: `<tr style="background:rgba(37,99,235,0.08)">` (синяя)
  - 2025: `<tr style="background:rgba(234,88,12,0.1)">` (оранжевая)
- Сортировка строк: от свежих к старым (Июнь → Январь текущего года, затем Декабрь → Январь предыдущего года)
- Колонка «Банк»: все банки кроме Сбера, отсортированные по убыванию количества жалоб
  - Формат: `Банк(N), Банк2(N)` (если N=1, можно без скобок)
- Если жалоб нет — нули с цветом `color:var(--text-faint)`, в колонке «Банк» — `—`
- Сбер уже в отдельной колонке, в колонку «Банк» его не включать

**Секция «Хронология»:**
- 4 колонки: Дата | Банк | Описание | Источник
- Для записей с непустым url: `<a href="url" target="_blank">текст описания</a>`
- Для записей без url: просто текст описания без ссылки
- Даты без дня (только месяц + год) — при сортировке считаются за 1-е число месяца

#### 5.4.5. Процесс обновления (cron-задача 08:50)

1. Прочитать `stats-inheritance-template.html` (шаблон)
2. Прочитать `katya-data.json` (единственный источник данных — katya-stats-data.md не используется)
3. Получить текущую дату: `TZ=Europe/Moscow date '+%Y-%m-%d'` и вчерашнюю: `date -d 'yesterday' '+%d %B %Y'`
4. Рассчитать все 6 секций из JSON (вчерашний день, месяц, годовые бары, таблица статистики, хронология)
5. Выполнить текстовые замены внутри шаблона — НЕ трогать HTML-теги, классы, атрибуты
6. Сохранить результат в `/tmp/stats-inheritance.html`
7. Запустить `publish-report.sh /tmp/stats-inheritance.html stats-inheritance.html`
8. Ответ в чат не требуется

## 7. Потоки данных (Data Flows)

### 7.1. Основной поток (ежедневно 08:30)

```
[08:30] Cron-триггер → Isolated-сессия OpenClaw
    ↓
[SearXNG] 11 curl-запросов к localhost:8888/search
    ↓
[web_fetch] Чтение каждого найденного URL
    ↓
[LLM (DeepSeek V4 Flash)] Фильтрация + группировка + категоризация
    ↓
┌─────────────────────────────────────┐
│           fork: 2 пути               │
├─────────────────────────────────────┤
↓                                   ↓
Telegram (Катя)                   katya-stats-data.md
08:30                              (копия данных)
    ↓
[08:50] Cron → stats-inheritance.html → publish-report.sh → GitHub Pages
```

### 7.2. Резервный поток (08:45, при сбое основного)

```
[08:45] Cron-триггер → Другая isolated-сессия (резервная)
    ↓
[SearXNG] Те же 11 запросов (повтор)
    ↓
[web_fetch] + [LLM] фильтрация
    ↓
Telegram (Катя)
```

### 7.3. Сбор архивных данных (разовый)

```
search_inheritance.py
    ↓
4 фазы:
  Фаза 1: 14 запросов × 2025 × 2 страницы
  Фаза 2: 14 запросов × 2026 × 2 страницы
  Фаза 3: 4 дополнительных запроса (цитатные)
  Фаза 4: Помесячные глубинные запросы
    ↓
scrape.js + generate.py → katya-data.json (77 записей)
```

## 8. Правила бизнес-логики (Business Rules)

| ID | Правило | Описание |
|----|---------|----------|
| BR-01 | **Только вчера** | Ежедневный поиск — только за вчерашний день. Никогда за сегодня |
| BR-02 | **Жёсткое отсечение РКЛ** | Жалобы про реестр контролируемых лиц (РКЛ/115-ФЗ/антиотмывочный) без связи с наследством — всегда исключаются |
| BR-03 | **Схлопывание дублей** | Одна жалоба = одна запись в katya-data.json, даже если найдена несколькими запросами |
| BR-04 | **Схлопывание сообщений автора** | Если один пользователь пишет несколько раз — одна жалоба |
| BR-05 | **URL-ссылки** | Если у жалобы есть URL — показывать как ссылку. Если нет — показывать полный текст description |
| BR-06 | **Цвет счётчика** | 0 жалоб за день = зелёный (>0 = красный) |
| BR-07 | **Отсутствие = не писать** | Если за день нет жалоб — не писать строку статистики в katya-stats-data.md |

## 10. Критерии приёмки (Acceptance Criteria)

### 10.1. Ежедневные проверки

- [ ] AC-01: Катя получает сводку в Telegram до 08:45 (включительно с резервом)
- [ ] AC-02: В сводке нет жалоб по 115-ФЗ/РКЛ без наследства
- [ ] AC-03: В сводке нет дублей
- [ ] AC-04: stats-inheritance.html опубликован на GitHub Pages

### 10.2. Качество данных

- [ ] AC-05: У каждой жалобы проставлен банк
- [ ] AC-06: У каждой жалобы указан источник (banki.ru, otzovik.com, и т.д.)
- [ ] AC-07: Счётчики на stats-inheritance.html складываются корректно
- [ ] AC-08: Динамика месяц к месяцу рассчитана верно

### 10.3. Надёжность

- [ ] AC-09: При ошибке основного запуска резервный отрабатывает
- [ ] AC-10: При 2+ ошибках подряд Кирилл получает алерт
- [ ] AC-11: Система не падает при пустом результате (0 жалоб за день)
