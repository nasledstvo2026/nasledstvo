# Бизнес-требования: Мониторинг жалоб по наследству в банках РФ

**Проект:** Цифровой продукт «Мониторинг жалоб»
**Версия:** 4.0
**Дата:** 02.07.2026
**Автор:** Лунт (Business Analyst)
**Статус:** Утверждено (по факту реализации)

---

## 1. Описание проблемной области

### 1.1. Предметная область
Наследование денежных средств в банках РФ. После смерти владельца вклада/счёта наследники сталкиваются с отказами, задержками и бюрократическими препятствиями при получении причитающихся им денежных средств.

### 1.2. Ключевые участники
- **Наследники** — граждане, вступающие в права наследования денежных средств на банковских счетах умерших родственников
- **Банки РФ** — Сбербанк, ВТБ, Т-Банк, ПСБ, Газпромбанк, Альфа-Банк и др. — организации, обязанные выдать наследственные средства
- **Площадки-источники** — banki.ru, otzovik.com, pikabu.ru, sravni.ru — места публикации жалоб
- **Катя** — Product Owner, ежедневно мониторит жалобы наследников, анализирует статистику по банкам-нарушителям, формирует на основе данных рабочие обращения и отчёты

### 1.3. Типовые проблемы (по жалобам)
- Отказ банка в выдаче вклада наследнику
- Задержка выдачи на недели и месяцы
- Требование избыточных документов (сверх установленных законом)
- Навязывание услуг нотариуса/адвоката «партнёра» банка
- Блокировка счетов по 115-ФЗ без связи с наследством
- Необоснованное удержание комиссий
- Отсутствие информации у сотрудников о процедуре наследования

### 1.4. Масштаб проблемы
- **77 жалоб в базе** (14.01.2025 — 30.06.2026)
- **19 банков** фигурируют в жалобах
- **Топ-5 банков-нарушителей:** Сбербанк, ВТБ, Т-Банк, ПСБ, Газпромбанк
- **4 площадки-источника:** banki.ru, otzovik.com, pikabu.ru, sravni.ru

### 1.5. Описание проблемы
До создания продукта Катя тратила часы на ручной поиск жалоб по разрозненным площадкам. Отсутствовала системная статистика, часть жалоб пропускалась, не было возможности отслеживать динамику по банкам-нарушителям.

## 2. Описание бизнес-цели

### 2.1. Основная цель (Business Goal)
**BG-1:** Автоматизировать для Кати ежедневный мониторинг жалоб по наследованию в банках РФ, исключив ручной поиск.

**BG-2:** Обеспечить для Кати формирование статистик по жалобам по наследству в банках (системная аналитика).

### 2.2. Составляющие цели
| ID | Составляющая цели | Как измеряем |
|----|-------------------|--------------|
| BG-1.1 | Катя получает готовую сводку жалоб в Telegram каждое утро без ручного поиска | Сводка доставляется ≥ 95% рабочих дней |
| BG-1.2 | Система отфильтровывает нерелевантные жалобы (РКЛ, 115-ФЗ, кредиты без наследства) | Precision фильтрации ≥ 90% |
| BG-2.1 | Система ведёт статистику по банкам, месяцам, динамике | stats-inheritance.html обновляется ежедневно |
| BG-2.2 | Система устойчива к сбоям и восстанавливается без участия Кати | Резервный запуск при сбое основного |

### 2.3. Единственный пользователь
**Катя** — Product Owner. Всё, что делает продукт, делается для Кати:
- Сводка жалоб в Telegram → для Кати
- Статистическая страница на GitHub Pages → для Кати
- Автоматический сбор, фильтрация, категоризация → для Кати
- Техническая поддержка и алерты об ошибках → для стабильной работы продукта Кати
- Системная аналитика → для Кати

### 2.4. Ценность
- Экономия времени: с 30+ минут ручного поиска до 0 (сводка приходит сама)
- Системная статистика: какие банки чаще нарушают, какие типы проблем преобладают
- База данных: накопленные жалобы для анализа и возможных обращений

## 8. Потоки данных (Data Flows)

### 8.1. Основной поток (ежедневно 08:30)

```
[Катя спит / занимается делами]
    ↑
[08:30] Cron-триггер → Isolated-сессия OpenClaw
    ↓
[SearXNG] 11 curl-запросов к localhost:8888/search
    ↓
[web_fetch] Чтение каждого найденного URL
    ↓
[LLM (DeepSeek V4 Flash)] Фильтрация + группировка + категоризация
    ↓
┌─────────────────────────────────────┐
│           fork: 2 пути               │
├─────────────────────────────────────┤
↓                                   ↓
Telegram (Катя)                   stats-inheritance-template
08:30                              → GitHub Pages (08:50)
```

> **Пользовательская история:** Катя просыпается и видит в Telegram готовую сводку. Ей не нужно ничего искать — система уже отфильтровала, сгруппировала по банкам, убрала дубли и нерелевантные жалобы.

### 8.2. Резервный поток (08:45, при сбое основного)

```
[08:45] Cron-триггер → Резервная isolated-сессия
    ↓
[SearXNG] Те же 11 запросов (повтор)
    ↓
[web_fetch] + [LLM] фильтрация
    ↓
Telegram (Катя)
```

> **Пользовательская история:** Катя всегда получает сводку, даже если основной запуск дал сбой. Техподдержка автоматически оповещается о проблеме.

### 8.3. Сбор архивных данных (разовый)

```
search_inheritance.py
    ↓
4 фазы:
  Фаза 1: 14 запросов × 2025 × 2 страницы
  Фаза 2: 14 запросов × 2026 × 2 страницы
  Фаза 3: 4 дополнительных запроса (цитатные)
  Фаза 4: Помесячные глубинные запросы
    ↓
katya-data.json (77 записей)
```

## 9. Правила бизнес-логики (Business Rules)

| ID | Правило | Описание |
|----|---------|----------|
| BR-01 | **Только вчера** | Ежедневный поиск — только за вчерашний день. Никогда за сегодня |
| BR-02 | **Жёсткое отсечение РКЛ** | Жалобы про реестр контролируемых лиц (РКЛ/115-ФЗ/антиотмывочный) без связи с наследством — всегда исключаются |
| BR-03 | **Схлопывание дублей** | Одна жалоба = одна запись в katya-data.json, даже если найдена несколькими запросами |
| BR-04 | **Схлопывание сообщений автора** | Если один пользователь пишет несколько раз — одна жалоба |
| BR-05 | **URL-ссылки** | Если у жалобы есть URL — показывать как ссылку. Если нет — показывать полный текст description |
| BR-06 | **Цвет счётчика** | 0 жалоб за день = зелёный (>0 = красный) |
| BR-07 | **Отсутствие = не писать** | Если за день нет жалоб — не писать строку статистики в katya-stats-data.md |
